


## 一、引言：问题的本质

当一个大语言模型被切分到多张 GPU 上运行时，每一次 forward pass 都会在计算（GEMM、Attention、RMSNorm 等）和通信（AllReduce、AllGather、ReduceScatter 等）之间反复交替。传统系统将二者视为完全独立的阶段——先完成本地计算，再启动跨卡通信——这种"先算后传"的串行范式在模型规模和 GPU 数量不断增长的今天，已经成为了显著的性能瓶颈。

本报告完整梳理"编译式优化从单卡计算优化扩展到分布式计算+通信联合优化"这一技术演进脉络，覆盖硬件互联、通信库、框架层编译优化、以及极致的 MegaKernel 融合四个层次。

## 二、硬件基础：GPU 互联的演进

### 2.1 PCIe 时代的瓶颈

在 NVLink 出现之前，GPU 之间的通信完全依赖 PCIe 总线。PCIe 3.0 x16 仅提供约 16 GB/s 的双向带宽，对于需要在每一层 Transformer 都执行 AllReduce 的张量并行（Tensor Parallelism）而言，通信延迟足以抵消多卡并行带来的计算加速。更关键的是，PCIe 通信必须经由 CPU 中转，引入了额外的延迟和 CPU 开销。

### 2.2 NVLink 的四代演进

NVIDIA 从 2016 年的 Pascal 架构开始引入 NVLink，其演进轨迹清晰地反映了对通信带宽的持续追求：

NVLink 1.0（Pascal P100, 2016）提供每 GPU 160 GB/s 的总带宽，首次实现 GPU 间直接高速互联，绕过 PCIe。NVLink 2.0（Volta V100, 2018）将带宽提升至 300 GB/s，并支持 GPU-GPU 之间的 cache coherence。NVLink 3.0（Ampere A100, 2020）进一步提升至 600 GB/s，并首次引入 NVSwitch 芯片，使得 DGX A100 中的 8 张 GPU 可以实现全互联拓扑，每对 GPU 之间都有等价的带宽。NVLink 4.0（Hopper H100, 2022）达到 900 GB/s，第二代 NVSwitch 使 8 张 GPU 全互联带宽进一步提升。更重要的是，H100 引入了 NVLink-SHARP（Scalable Hierarchical Aggregation and Reduction Protocol），允许在 NVSwitch 硬件层面直接执行归约操作，将 AllReduce 的部分计算从 GPU 卸载到网络交换芯片。

### 2.3 NVLink 5.0 与 Blackwell 架构

2024 年发布的 Blackwell 架构将 NVLink 推向了第五代。GB200 NVL72 系统通过 NVLink 5.0 将 72 张 Blackwell GPU 互联在一个统一的 NVLink 域中，总 GPU 间通信带宽达到 130 TB/s，单 GPU 的 NVLink 带宽达到 1.8 TB/s。这意味着在机柜级别，72 张 GPU 可以像一张巨大的虚拟 GPU 一样协作，任意两张 GPU 之间的通信带宽一致且极高。

这种硬件演进为"计算+通信联合优化"提供了物理基础：当互联带宽足够高时，通信不再是不可掩盖的固定成本，而是可以与计算流水线化、重叠执行的资源。

### 2.4 NVLink 的可编程性：从黑箱到用户可控

早期 NVLink 通信只能通过 NCCL 库发起，对用户是黑箱。近年来，NVIDIA 逐步开放了 NVLink 的可编程性。NVSHMEM 允许 GPU kernel 内部直接通过 load/store/atomic 指令访问远程 GPU 内存，绕过 NCCL 的 kernel 级别同步机制。这一能力是后续 MegaKernel 技术的关键基石——只有当通信操作可以在 kernel 内部以指令级粒度发起时，计算和通信的细粒度融合才成为可能。

## 三、通信库层：从 NCCL 到 Symmetric Memory

### 3.1 NCCL 的经典模型

NCCL（NVIDIA Collective Communications Library）是 GPU 集群通信的事实标准。其核心设计理念是将通信和归约计算统一在一个 GPU kernel 中执行，自动检测网络拓扑并选择最优通信算法（Ring、Tree、NVLS 等）。从用户视角看，一次 `ncclAllReduce` 调用就是一个完整的 kernel launch，它在 GPU 的 SM 上执行数据搬运和归约运算。

这种设计的问题在于：NCCL 的 AllReduce kernel 会占用 GPU 的 SM 资源，而在 kernel 执行期间，这些 SM 无法被用于模型计算。NCCL issue #403 中记录了典型案例——`ncclAllReduceRingLLKernel` 占用了 13.4% 的 CUDA 资源，导致模型计算被显著拖慢。

### 3.2 NCCL 2.27：Symmetric Memory 与低延迟 kernel

2025 年 7 月发布的 NCCL 2.27 是通信库层的重大里程碑。其核心创新是引入了 Symmetric Memory 支持：当各 GPU 上的通信缓冲区被分配在相同的虚拟地址时，NCCL 可以使用专门优化的低延迟 kernel，大幅降低同步开销。基准测试表明，小消息 AllReduce 延迟最高降低了 9 倍。在 B200 上，2 GPU 的小消息 AllReduce 延迟仅约 9 微秒。

NCCL 2.27 同时引入了对 NVLink SHARP 和 InfiniBand SHARP 的联合支持，使得 AllGather 和 ReduceScatter 可以卸载到网络交换芯片执行。传统的 Ring-based 实现需要消耗 16 个以上 SM，而 SHARP 支持将 SM 占用降低到 6 个以内。这意味着通信占用的计算资源大幅减少，为计算-通信重叠创造了更多空间。

### 3.3 PyTorch SymmetricMemory：面向框架层的可编程通信

PyTorch 团队在 2024 年底引入了 `torch.distributed._symmetric_memory` 抽象。这一抽象在各 GPU 之间创建一个全局地址空间，允许通过 CUDA 的 P2P 机制直接进行 load/store 操作。关键优势在于：数据传输可以由 Copy Engine 处理，完全不占用 SM，从而消除了通信 kernel 与计算 kernel 之间的 SM 资源竞争。

这个设计直接支撑了 PyTorch 的 Async Tensor Parallelism 实现。TorchTitan 团队基于 SymmetricMemory 构建了 `torch.ops.symm_mem.fused_all_gather_matmul` 和 `torch.ops.symm_mem.fused_matmul_reduce_scatter` 两个融合算子，实现了 AllGather 与 GEMM、GEMM 与 ReduceScatter 的流水线重叠执行。

## 四、框架层：从 Megatron-LM 到 vLLM 的编译式通信优化

### 4.1 Megatron-LM：Tensor Parallelism 和 Sequence Parallelism 的起源

理解通信优化的历史脉络，需要从 Megatron-LM 的两篇关键论文讲起。

2019 年，NVIDIA 发表的 Megatron-LM 论文首次系统性地提出了 Transformer 模型的张量并行方案。其核心思想是将 MLP 层的权重矩阵沿列（column-parallel）和行（row-parallel）进行切分，每张 GPU 计算部分结果，然后通过 AllReduce 同步。在标准的 Megatron TP 方案中，每个 Transformer 层需要执行 2 次 AllReduce（MLP 一次、Attention 一次）。AllReduce 的通信量为 $2 \times (p-1)/p \times hidden_size \times seq_len \times batch_size$（其中 $p$ 为 TP 度），这在跨服务器时开销巨大，因此 Megatron-LM 将 TP 限制在节点内、依赖 NVLink 执行。

2022 年，NVIDIA 发表了 “Reducing Activation Recomputation in Large Transformer Models”（MLSys 2023），正式提出了 Sequence Parallelism（SP）概念。其关键洞察是：在 TP 中，LayerNorm 和 Dropout 等操作是在完整的激活值上独立执行的，不需要跨 GPU 通信；如果把 AllReduce 分解为 ReduceScatter + AllGather，就可以在 ReduceScatter 之后让每张 GPU 仅对自己负责的 sequence 分片执行 LayerNorm，然后再通过 AllGather 恢复完整张量供下一个并行计算使用。这一变换将通信模式从 AllReduce 改为 ReduceScatter + AllGather，虽然通信总量不变，但创造了计算与通信重叠的结构性机会——ReduceScatter 可以与上一层的计算重叠，AllGather 可以与下一层的计算重叠。

### 4.2 Async Tensor Parallelism：打破计算与通信的抽象壁垒

2024 年的论文 “Breaking the Computation and Communication Abstraction Barrier in Distributed Machine Learning Workloads” 以及相关的并行工作系统性地提出了 Async TP 的概念：通过将通信和计算算子分解为更细粒度的子操作，使二者可以交错执行。

以 AllGather + GEMM 为例：传统方式先完成 AllGather 收集所有分片，再启动完整的 GEMM 计算。Async TP 将 AllGather 分解为多步 recv 操作，将 GEMM 分解为多个 sub-GEMM，每收到一个分片就立即启动对应的 sub-GEMM，从而实现通信与计算的流水线重叠。对于 GEMM + ReduceScatter 方向同理：GEMM 每完成一个 tile 的输出，就立即对该 tile 发起 ReduceScatter。

PyTorch TorchTitan 团队在 2024 年 9 月公开了基于 SymmetricMemory 的 Async TP 实现，并在 Llama3 8B 上观察到前向传播最高 29% 的加速、端到端最高 8% 的加速；在 Llama3 70B 上，前向最高 20% 加速、端到端 8% 加速。

实现中遇到的关键工程挑战包括：SM 资源竞争——NCCL 的 send/recv 使用 SM 来通过 NVLink 搬运数据，减少了 GEMM 可用的 SM 数量，且 cuBLAS 的 wave quantization 效应会放大这一损失。TorchTitan 的解决方案是使用 Copy Engine 而非 SM 来执行 NVLink 数据传输。Wave Quantization 恶化——将一个大 GEMM 分解为多个 sub-GEMM 后，每个 sub-GEMM 的 tile 数量减少，导致最后一个 wave 的利用率更低。解决方案是采用"交替流"（alternating-stream）策略，让当前 sub-GEMM 的不完整 wave 与下一个 sub-GEMM 重叠。

### 4.3 vLLM 的编译式通信融合 Pass

vLLM 将上述理念通过 `torch.compile` 的 Inductor Pass 机制实现为一套系统化的编译期优化。这是当前推理引擎中最完整的"编译式计算+通信联合优化"框架。

vLLM 的通信融合包含三个递进层次：

**第一层：AllReduce + RMSNorm 融合（`fuse_allreduce_rms`）**。这是最直接的融合，将 TP 中的 AllReduce 与紧随其后的残差连接和 RMSNorm 合并为一个 FlashInfer/TRT-LLM 融合 kernel。其收益在于避免了 AllReduce 结果写回全局内存、再被 RMSNorm 读取的多余内存往返。该融合仅在小 batch 时有效（因为此时 kernel launch 开销占比高），要求 SM90+（Hopper/Blackwell）硬件，支持 BF16/FP8/NVFP4 多种精度。端到端加速约 5-20%。

**第二层：Sequence Parallelism 变换（`enable_sp`）**。这是一个图级别的重构 Pass，将 `AllReduce → RMSNorm` 模式替换为 `ReduceScatter → local RMSNorm → AllGather`。这个变换本身并不直接提升性能，但它是 Async TP 融合的前置条件——只有将 AllReduce 分解为 ReduceScatter + AllGather 后，才能分别与上下游的 GEMM 进行融合。该 Pass 仅在大 batch 场景（token 数量超过阈值）下激活，目前自动配置仅限 H100（SM90）且 hidden_size >= 8192 的模型。

**第三层：AsyncTP GEMM + Collective 重叠（`fuse_gemm_comms`）**。在 SP 变换完成后，这个 Pass 将 `GEMM → ReduceScatter` 融合为 `fused_matmul_reduce_scatter`，将 `AllGather → GEMM` 融合为 `all_gather_matmul`，底层使用 `torch.ops.symm_mem` 的 SymmetricMemory 原语实现通信与计算的流水线重叠。支持 FP8 scaled 变体。端到端加速约 7-10%。

这三个 Pass 的设计体现了"编译式优化"的核心理念：模型代码不需要任何修改，所有优化在编译期通过图模式匹配和变换自动完成。用户只需设置优化级别（如 `-O2`），框架自动根据硬件能力和模型特征选择合适的融合策略。

### 4.4 TokenWeave：通信+计算融合的另一范式

微软研究院在 2025 年 5 月提出的 TokenWeave 方案代表了另一种融合思路。TokenWeave 的核心创新包括：Token-Splitting 技术，将推理 batch 中的 token 分为两个大致等量的子集，以 wave-aware 的方式调度，使一个子集的 GEMM 计算与另一个子集的通信重叠。Fused AllReduce + RMSNorm + Residual kernel，在一个自定义 kernel 中同时完成 AllReduce 归约、残差连接和 RMSNorm 归一化，避免了三次独立 kernel launch 的开销。即使在 NVLink 高速互联下，分布式推理的通信开销仍可高达 20%，TokenWeave 将这一开销大幅降低。

## 五、Kernel 层：从 Kernel Fusion 到 MegaKernel

### 5.1 传统 Kernel Fusion 的局限

传统的 kernel fusion 主要在单 GPU 上优化相邻算子，典型案例包括 RMSNorm + Quantization、SiLU + Mul + Quantization 等。这些融合消除了中间结果对全局内存的读写，但有一个根本限制：它们的作用范围仅限于少数相邻算子，不涉及跨算子流水线和跨 GPU 通信。

PyTorch、JAX、TVM 等框架使用规则驱动的启发式方法进行 fusion 决策，Mirage、TASO 等系统通过编译器超优化自动发现 fusion 规则。但这些工具都只能融合小的局部算子组合，生成一个忠实实现复杂张量程序的单一 kernel 在计算上是困难的。

### 5.2 Stanford HazyResearch：手工 MegaKernel 的突破

2025 年 5 月，Stanford HazyResearch 团队发布了标志性的工作"Look Ma, No Bubbles!"，将 Llama-1B 的整个 forward pass 融合为单个 MegaKernel。

其动机来自一个令人震惊的观察：在 batch size 1 的 decode 场景下，vLLM 和 SGLang 仅利用了 H100 约 50% 的内存带宽。原因是每次 forward pass 约有 100 个独立 kernel launch，每个 kernel 之间存在 setup/teardown 的"memory pipeline bubble"。即使使用 CUDA Graph，每次 kernel launch 仍有约 1.3 微秒的不可消除延迟。按照 7 个 kernel/层 × 16 层、每个 kernel 5 微秒的 stall 估算，理论 1350 次/秒的 forward pass 速率会降至 770 次/秒。

HazyResearch 的解决方案是一个基于 on-GPU interpreter 的 MegaKernel 架构。每个 SM 接收一个预先在 Python 侧编排好的指令序列，包含 fused RMSNorm+QKV+RoPE、Attention、O-projection+residual、fused RMSNorm+up-gate+SiLU、down-projection+residual 等复合指令。关键技术创新包括：共享内存分页（将 H100 的 213KB 共享内存分为 13 个 16KB 页面，指令显式申请和释放页面，使下一条指令可以在上一条指令完成前就开始加载权重）、Counter-based 同步（在全局内存中维护计数器数组替代 kernel barrier，实现指令间细粒度依赖控制）、以及分块流水线（将 MLP 的中间状态分为四个 chunk，每个 chunk 有独立计数器，下游指令只需等待其输入 chunk 完成）。

结果是在 H100 上实现了 78% 的内存带宽利用率，比 vLLM 快约 2.5 倍，比 SGLang 快约 1.5 倍。在 B200 上，单次 forward pass 时间低于 680 微秒，首次将 1B+ 参数 16 位模型的 forward pass 推进到亚毫秒级别。

### 5.3 HazyResearch 的多 GPU MegaKernel 扩展

2025 年 9 月，HazyResearch 将 MegaKernel 扩展到了多 GPU 场景，发布了支持 8-GPU 张量并行的 Llama-70B 推理 MegaKernel。这是"编译式优化扩展到通信层"的手工实现范例。

在多 GPU 场景下，MegaKernel 统一处理三种重叠：Intra-SM 重叠（消除所有 kernel launch，通过指令调度器实现内存传输与计算的软件流水线）、Cross-SM 重叠（不同 SM 可以同时执行同一模型的不同部分，通过动态指令分配实现负载均衡）、以及 Cross-GPU 重叠（直接在 kernel 内部通过 remote load/store 发起跨 GPU 通信，将通信分解为细粒度传输并与本地计算交错执行，关键路径永远不会 stall 在 CPU 上）。

传统多 GPU 系统使用 CUDA stream 来重叠通信和计算——一个 stream 运行 NCCL AllReduce，另一个处理下一层计算——这需要 CPU 调度器精确管理依赖和时序。MegaKernel 方案完全消除了这种 CPU 介入，所有调度决策在 kernel 内部完成。集成到 tokasaurus 推理引擎后，在 ShareGPT 数据集的 65536 个 prompt 上比 SGLang 快约 22%。

### 5.4 MPK：编译器自动生成 MegaKernel

CMU 的 Zhihao Jia 团队在 2025 年 12 月发布了 Mirage Persistent Kernel（MPK），这是第一个能够自动将多 GPU 模型推理编译为高性能 MegaKernel 的编译器和运行时系统。

MPK 的核心创新是 SM-level graph representation（ttGraph）。传统的计算图以算子为节点、张量为边，而 ttGraph 以 SM 级别的 task 为节点、event 为同步点，能表达比 kernel 级别粒度更细的依赖关系。例如，MatMul 后接 AllReduce 时，AllReduce 的每个 task 实际上只依赖 MatMul 的一个对应 task，而非整个 MatMul 算子。ttGraph 通过在相关 task 对之间插入细粒度 event，使得 MatMul 的部分 SM 可以与 AllReduce 的部分 SM 并行执行。

MPK 编译器的工作流程包括：算子分解（将每个算子按输出张量的 tile 分配为 SM 级别 task）、依赖分析（精确枚举 task 对之间的数据依赖）、Event Fusion（消除冗余同步点）、ttGraph 归一化（确保每个 task 恰好有一个依赖 event 和一个触发 event）、以及 ttGraph 线性化（通过 BFS 确保被同一 event 触发的所有 task 具有连续索引）。

MPK 的 in-kernel parallel runtime 将 GPU 的 SM 划分为 worker 和 scheduler。Worker 维护独立的 task 队列并按 FIFO 执行；Scheduler 以 warp 为单位组织，监控 event 激活状态并将就绪的 task 分派给空闲的 worker。Task 发射采用 JIT（just-in-time）和 AOT（ahead-of-time）混合策略：JIT 在依赖满足后才分配 task，支持动态负载均衡（对 Attention 这类执行时间数据相关的算子尤为关键）；AOT 预先将 task 放入 worker 队列，减少调度延迟。

评估显示，MPK 在 A100、H100、B200 三代 GPU 上对五个广泛使用的模型实现了 1.0-1.7 倍的加速，将 LLM 推理性能推进到接近硬件理论极限。

### 5.5 FlashDMoE/FlashMoE：MoE 场景的通信融合

2025 年 6 月发表的 FlashDMoE 是另一个将计算和通信融合到单个 persistent kernel 的标志性工作，专注于 Mixture-of-Experts（MoE）场景。MoE 模型在 Expert Parallelism 下需要执行 All-to-All 通信来路由 token 到不同 GPU 上的 expert，传统实现中通信与计算完全串行。FlashDMoE 使用 tile-level parallelism 和 GPU-initiated communication（基于 NVSHMEM），将 expert 计算和跨 GPU token 路由融合在一个 kernel 中，实现了最高 9 倍的 MoE 性能提升。

## 六、技术演进的完整脉络

将上述各层次的发展整理为时间线，可以清晰看到技术演进的逻辑：

**第一阶段：单算子优化（~2020 年前）**。此阶段的优化集中在单个 kernel 的性能上。cuBLAS 提供高度优化的 GEMM 实现，FlashAttention 将 Attention 的多个操作融合为 IO-aware 的单个 kernel。通信和计算是完全独立的世界——NCCL 负责通信，计算 kernel 各自优化。

**第二阶段：相邻算子融合（2020-2022）**。PyTorch 的 `torch.compile`、TVM、XLA 等编译器开始支持将相邻的 element-wise 算子和 reduce 算子融合为单个 kernel，消除中间内存访问。但融合范围仅限于计算算子，不涉及通信。

**第三阶段：Sequence Parallelism 的提出（2022）**。NVIDIA 的 SP 论文将 AllReduce 分解为 ReduceScatter + AllGather，从算法层面创造了通信与计算重叠的结构性机会。这是通信优化从"硬件/通信库层面提升带宽"转向"算法/框架层面创造重叠机会"的关键转折点。

**第四阶段：Async TP 与框架层通信融合（2024-2025）**。PyTorch SymmetricMemory、TorchTitan 的 Async TP、vLLM 的编译式通信融合 Pass 将通信优化提升到框架编译器层面。通过 torch.compile 的图模式匹配自动识别 TP 模式，并重写为重叠版本，模型代码无需修改。AllReduce + RMSNorm 融合、GEMM + ReduceScatter 融合、AllGather + GEMM 融合成为标准化的编译 Pass。

**第五阶段：MegaKernel 与全融合（2025-至今）**。HazyResearch 的手工 MegaKernel 和 CMU 的 MPK 编译器将融合推向极致——整个模型的 forward pass（包括所有计算和跨 GPU 通信）被编译为单个 persistent kernel。kernel barrier 被完全消除，取而代之的是 SM 级别的细粒度 event 同步。计算和通信不再是两个需要被"重叠"的独立流，而是被编织在同一个 kernel 的 task graph 中，由 SM-level runtime 动态调度。

## 七、关键技术挑战与未来方向

### 7.1 当前挑战

**动态性问题**。当前 MegaKernel 和编译式通信融合主要面向静态或半静态场景。LLM 推理中的 batch size、sequence length 是动态变化的，这要求融合策略能够适应不同的工作负载特征。vLLM 的做法是维护多个编译后的 kernel 变体（piecewise compilation），根据当前的 token 数量选择合适的版本。MPK 通过 JIT task launch 部分解决了动态负载均衡的问题。

**SM 资源分配**。在通信和计算共享 SM 时，如何分配 SM 资源是一个开放问题。NCCL 的 SHARP 支持将 SM 需求从 16 个降低到 6 个，但对于没有 SHARP 硬件支持的场景，通信 kernel 对 SM 的占用仍然会降低计算 kernel 的性能。MegaKernel 方法通过统一调度解决了这个问题，但需要精心设计 task 调度策略。

**精度和正确性**。通信融合中的归约操作涉及浮点精度问题。NCCL 2.27 使用 FP32 累加器（NVLink SHARP 上 FP8 使用 FP16 累加器），但在 MegaKernel 内部的自定义通信实现中，精度控制需要更加谨慎。

**通用性与可移植性**。手工 MegaKernel（如 HazyResearch 的实现）需要大量的 CUDA 专家知识，且对不同模型架构需要重新实现。MPK 试图通过编译器自动化解决这一问题，但目前仍处于早期阶段，支持的算子类型和硬件平台有限。vLLM 的编译 Pass 方案在通用性上更好，因为它在 `torch.compile` 框架内工作，但融合的粒度不如 MegaKernel 极致。

### 7.2 未来方向

**跨节点通信融合**。当前的融合优化主要针对节点内（NVLink 互联）的 TP 通信。随着 NVIDIA 的 Direct NIC 支持（PCIe Gen6 直连 CX8 网卡，绕过 CPU 瓶颈）和跨数据中心通信的早期支持（NCCL 2.27），将融合扩展到跨节点的 Pipeline Parallelism 和 Data Parallelism 通信是自然的下一步。

**硬件-软件协同设计**。NVLink SHARP 允许在交换芯片上执行归约操作，这是硬件层面直接支持通信融合的范例。未来可能出现更多专用硬件——例如在 NVSwitch 上支持更复杂的通信-计算融合操作，或者 GPU 架构层面提供更高效的跨 SM 同步原语（当前 CUDA 的异步 barrier 每次操作约 60 纳秒，在 MegaKernel 中积累成为可观的开销）。

**通用 MegaKernel 编译器**。MPK 代表了这一方向的初步探索。理想的通用编译器应该能够从 PyTorch 模型定义自动生成覆盖全部计算和通信的 MegaKernel，支持任意模型架构、任意并行策略、任意硬件拓扑。这需要解决 SM-level 调度的搜索空间爆炸问题、跨多种硬件后端的代码生成问题、以及动态工作负载的运行时适配问题。

## 八、总结

从 Megatron-LM 的 TP AllReduce 到 vLLM 的编译式 Async TP Pass，再到 MPK 的全 forward pass MegaKernel，我们见证了编译式优化从"单卡计算优化"到"分布式计算+通信联合优化"的完整演进。这一演进的驱动力来自三个方面的同步进展：硬件层面——NVLink 带宽从 160 GB/s 增长到 1.8 TB/s，NVSwitch 从不存在到支持 72 GPU 全互联，SHARP 硬件归约能力的引入，NVLink 可编程性的开放；通信库层面——NCCL 从黑箱 API 演进到提供 Symmetric Memory 低延迟 kernel，PyTorch SymmetricMemory 使框架能直接控制 NVLink 通信方式；编译器/框架层面——从手工优化到 torch.compile 自动图匹配和重写，从局部算子融合到全模型 MegaKernel 编译。

这三个层次的协同演进正在模糊"计算"与"通信"之间的传统边界。在 MegaKernel 的视角下，AllReduce 不再是一个独立的通信原语，而是模型计算图中的一组 SM-level task，可以与矩阵乘法的 tile 级任务交错调度和执行。这种融合不仅是性能优化的技术手段，更代表了分布式 AI 系统编程范式的根本转变。
