

# 大模型训练与推理通信技术全景图


## 〇、第一性原理：为什么通信是核心矛盾

大模型训推的本质，是将一个**无法放进单个芯片**的计算任务，拆散到成百上千个计算单元上，让它们**协同**完成。协同就意味着通信。一个简单的公式决定了一切：

```
有效算力 = 峰值算力 × 计算效率
计算效率 ≈ 计算时间 / (计算时间 + 通信时间 + 同步等待时间)
```

当单卡算力从A100的312 TFLOPS（BF16）飙升到B200的9 PFLOPS（FP4），而互联带宽的增长速度远远跟不上算力增长时，系统从**Compute-bound**转变成**Communication-bound**。模型越大、GPU越多，通信墙越高。这就是这整份技术图景存在的根本原因。

## 一、五层互联模型：空间视角下的通信全景

要理解整个通信体系，第一件事是建立**空间层次感**。数据从一个计算单元流向另一个计算单元，根据物理距离的不同，经过的互联技术完全不同：

```
┌─────────────────────────────────────────────────────────────────┐
│  第5层：集群间互联（跨数据中心）                                      │
│  技术：WAN、专线、光传输网络                                          │
│  距离：公里级    带宽：Tbps级（骨干）   延迟：毫秒级                     │
├─────────────────────────────────────────────────────────────────┤
│  第4层：机间互联（节点间 / 跨机通信）                                  │
│  技术：InfiniBand、RoCE v2、Ultra Ethernet                        │
│  距离：1~100米   带宽：200~800 Gbps/端口  延迟：1~5 μs               │
│  ★ 这一层是大模型分布式训练的【主战场】                                │
├─────────────────────────────────────────────────────────────────┤
│  第3层：机内互联（节点内GPU-GPU / GPU-CPU）                           │
│  技术：NVLink/NVSwitch、PCIe、华为灵衢UB、CXL                      │
│  距离：厘米~米级  带宽：64 GB/s(PCIe) ~ 3.6 TB/s(NVLink6) 延迟：亚μs │
│  ★ 张量并行的核心依赖                                                │
├─────────────────────────────────────────────────────────────────┤
│  第2层：板内互联（芯片到芯片）                                         │
│  技术：NVLink-C2C、UCIe、HBM接口、片上网络(NoC)                      │
│  距离：毫米~厘米  带宽：900 GB/s ~ 1.8 TB/s   延迟：十纳秒级           │
├─────────────────────────────────────────────────────────────────┤
│  第1层：芯片内互联（Die内部）                                         │
│  技术：片上网络(NoC)、Crossbar、Ring Bus                            │
│  距离：微米~毫米  带宽：TB/s级   延迟：纳秒级                          │
└─────────────────────────────────────────────────────────────────┘
```

大模型训推重点关注的是**第3层（机内）和第4层（机间）**，这正好对应了你提到的所有关键技术：NVLink/灵衢UB负责第3层，InfiniBand/RoCE/RDMA负责第4层。

## 二、RDMA：一切高速通信的地基

### 2.1 传统通信的问题

传统TCP/IP协议下，数据从一个GPU到另一台机器的GPU，需要走完这条路：

```
GPU显存 → CPU内存(拷贝1) → 内核协议栈(TCP/IP处理) → NIC缓冲区(拷贝2) 
  → 网线 → 
NIC缓冲区 → 内核协议栈(拷贝3) → CPU内存(拷贝4) → GPU显存(拷贝5)
```

5次内存拷贝、多次CPU中断、完整的内核协议栈处理。每一步都消耗CPU周期和内存带宽，延迟高达**数十到数百微秒**。

### 2.2 RDMA的三大核心能力

RDMA（Remote Direct Memory Access）通过三个机制颠覆了这一切：

**零拷贝（Zero-Copy）**：应用程序注册的内存区域（Memory Region）被锁定在物理内存中，网卡直接从该区域读写数据，不需要在用户空间和内核空间之间拷贝。

**内核旁路（Kernel Bypass）**：应用程序通过用户态的Verbs API直接向网卡下发工作请求（Work Request），不需要经过操作系统内核，消除了系统调用和上下文切换的开销。

**协议卸载（Protocol Offload）**：可靠传输、重传、流控等协议逻辑全部在网卡硬件（RNIC）中执行，CPU完全不参与数据面处理。

RDMA之后的数据路径：

```
GPU显存 →(GPUDirect RDMA)→ RNIC → 网线 → RNIC →(GPUDirect RDMA)→ GPU显存
```

延迟降至**1~2微秒**，CPU参与度接近为零。

### 2.3 RDMA关键编程抽象

```
┌──────────────────────────────────────────────────┐
│              RDMA 编程模型                         │
│                                                    │
│  Protection Domain (PD)                            │
│    └── 安全隔离域，圈定可访问的资源                    │
│                                                    │
│  Memory Region (MR)                                │
│    └── 注册物理内存，获得lkey/rkey                    │
│    └── 远端通过rkey可直接读写（不通知对方CPU!）        │
│                                                    │
│  Queue Pair (QP)                                   │

│    └── Receive Queue (RQ)：接收侧                   │
│    模式：RC(可靠连接)/UC(不可靠连接)/UD(不可靠数据报)  │
│                                                    │
│  Completion Queue (CQ)                             │
```

### 2.4 RDMA的三条实现路径

```
               RDMA
              ╱  │  ╲
            ╱    │    ╲
          ╱      │      ╲
    InfiniBand  iWARP   RoCE
    (原生RDMA)  (TCP上)  (以太网上)
                         ╱    ╲
                     RoCE v1  RoCE v2
                     (L2)    (L3,可路由)
```

三条路径共享同一套Verbs API，上层应用（如NCCL）可以透明切换。但底层网络特性差异巨大，直接影响大模型训练的稳定性和性能。

### 2.5 GPUDirect技术族

这是NVIDIA为了让RDMA与GPU紧密配合而发展出的一系列技术：

```
GPUDirect技术演进：

2010  GPUDirect v1（Shared Pinned Memory）
      └─ CPU内存中开辟共享区，GPU和NIC都访问该区域
      └─ 仍有一次CPU内存拷贝

2011  GPUDirect P2P（Peer-to-Peer）
      └─ 同一PCIe域内GPU直接访问另一GPU显存
      └─ 通过PCIe总线，不经CPU内存
      └─ 受限于PCIe带宽（~32 GB/s per direction）

2013  GPUDirect RDMA (GDR)
      └─ 网卡直接读写GPU显存，完全绕过CPU内存
      └─ 跨节点GPU-GPU通信的核心
      └─ 要求NIC和GPU在同一PCIe root complex下

2019  GPUDirect Storage (GDS)
      └─ NVMe SSD直接与GPU显存交换数据
      └─ 用于模型checkpoint加载、大数据集读取

2025  GPUDirect TCPX/TCPXO
      └─ 面向云环境的GPU直通网络方案
```

### 3.3 IB为什么是AI训练的首选

**天然无损**：基于信用的流控（Credit-based Flow Control）在链路层就确保了发送方不会发出超过接收方缓冲区容量的数据。对比以太网上的RoCE需要额外部署PFC/ECN/DCQCN才能"模拟"无损，IB从协议设计上就避免了丢包。而RDMA协议对丢包极度敏感——一个丢包可能导致整个QP状态机崩溃，需要重建连接。

**自适应路由（Adaptive Routing）**：Quantum系列交换机支持动态路由，数据包可以根据链路负载实时选择最优路径，避免热点。

**SHARP（Scalable Hierarchical Aggregation and Reduction Protocol）**：这是IB独有的杀手级特性——在网计算。AllReduce操作不需要所有数据汇集到某个节点再分发，而是在交换机内部直接完成聚合计算。对于大模型训练中频繁的梯度聚合，这可以将通信量降低近一半。

**拥塞控制**：IB拥有成熟的硬件级拥塞控制机制，而不依赖软件层面。

## 四、RoCE：以太网上的RDMA

### 4.1 为什么需要RoCE

InfiniBand虽然性能卓越，但有三个问题：NVIDIA（Mellanox）独家垄断供应链、成本高昂、与现有以太网基础设施不兼容。RoCE（RDMA over Converged Ethernet）的目标就是在廉价且普遍部署的以太网上实现RDMA能力。

### 4.2 RoCE v1 vs RoCE v2

```
RoCE v1：
┌──────────────────────────────────┐
│ Ethernet Header │ IB GRH │ IB BTH │ Payload │ ICRC │ FCS │
└──────────────────────────────────┘
→ IB包直接封装在以太网帧中
→ L2协议，不能跨子网路由 ← 致命缺陷

RoCE v2：
┌──────────────────────────────────────────────────┐
│ Ethernet Hdr │ IP Hdr │ UDP Hdr │ IB BTH │ Payload │ ICRC │
└──────────────────────────────────────────────────┘
→ 用IP/UDP封装，可以跨L3路由  ← 关键改进
→ UDP目的端口4791
→ 当前事实标准
```

### 4.3 RoCE的核心难题：构建无损以太网

以太网是"Best-Effort"的——拥塞时可以丢包。但RDMA对丢包极度敏感。这形成了根本矛盾。RoCE的部署因此需要一整套"无损以太网"技术栈：

```
无损以太网技术栈：

┌─────────────────────────────────────────────────────────────┐
│                        应用层                                │
│   NCCL / 训练框架 ← 对底层无感，通过Verbs API调用            │
├─────────────────────────────────────────────────────────────┤
│                      传输层优化                              │
│   DCQCN (Data Center QCN)                                   │
│     └─ RoCE专用拥塞控制算法                                  │
│     └─ 收到ECN标记后降低发送速率                              │
│     └─ 基于令牌桶的速率控制                                  │
├─────────────────────────────────────────────────────────────┤
│                      网络层标记                              │
│   ECN (Explicit Congestion Notification)                    │
│     └─ 交换机队列深度超阈值时在IP头标记CE位                    │
│     └─ 端到端的拥塞信号                                     │
├─────────────────────────────────────────────────────────────┤
│                      链路层流控                              │
│   PFC (Priority Flow Control) — IEEE 802.1Qbb              │
│     └─ 逐跳（hop-by-hop）的暂停机制                          │
│     └─ 当接收缓冲区快满时，向上游发PAUSE帧                    │
│     └─ 按优先级（Priority）粒度暂停                          │
│     └─ ⚠️ PFC风暴/死锁是RoCE部署的头号噩梦                  │
├─────────────────────────────────────────────────────────────┤
│                      交换机配置                              │
│   QoS：划分RDMA流量到独立队列                                │
│   Buffer管理：精确配置共享/私有缓冲区                         │
│   Watchdog：检测并缓解PFC风暴                                │
└─────────────────────────────────────────────────────────────┘
```

### 4.4 RoCE vs InfiniBand：核心差异总结

```
维度             InfiniBand            RoCE v2
─────────────────────────────────────────────────
无损保障         原生（Credit-based）    需要PFC+ECN构建
拥塞控制         硬件级                 DCQCN（软硬结合）
路由             自适应路由              ECMP（等价多路径）
在网计算         SHARP ✓               ✗
供应商           NVIDIA独家              多厂商（开放生态）
交换机成本        高                    相对低
运维复杂度        低（即插即用）         高（需精细调优）
可扩展性         子网~数千节点           理论更大（L3路由）
大规模实践        NVIDIA DGX SuperPOD   Meta 24K GPU集群
```

### 4.5 新势力：Ultra Ethernet (UEC)

Ultra Ethernet Consortium于2023年成立，目标是为AI/HPC重新设计以太网传输层。核心改进包括：取消对PFC的依赖，使用新的无损传输协议；支持多路径传输和乱序交付；原生支持集合通信操作。UEC 1.0规范已于2025年发布，但硬件产品仍在早期阶段。

## 五、NVLink与NVSwitch：芯片级到机柜级的直连高速公路

### 5.1 为什么PCIe不够

PCIe是通用互联标准，但带宽增长跟不上GPU算力增长：

```
PCIe Gen3 x16: 32 GB/s    (2010年代)
PCIe Gen4 x16: 64 GB/s    (2020年)
PCIe Gen5 x16: 128 GB/s   (2023年)
PCIe Gen6 x16: 256 GB/s   (预计2025-2026年)

对比：
NVLink 4.0 (H100): 900 GB/s    = PCIe 5.0 的 7倍
NVLink 5.0 (B200): 1,800 GB/s  = PCIe 5.0 的 14倍
NVLink 6.0 (Rubin): 3,600 GB/s = PCIe 5.0 的 28倍
```

这就是NVLink存在的理由：在GPU之间提供远超PCIe的带宽和远低于PCIe的延迟。

### 5.2 NVLink代际演进全谱

```
代际      架构      每GPU带宽   链路数  单链路速率   关键特性
────────────────────────────────────────────────────────────────
NVLink 1  Pascal    160 GB/s    4     40 GB/s     首代，GPU-GPU直连
NVLink 2  Volta     300 GB/s    6     50 GB/s     引入缓存一致性(CPU-GPU)
NVLink 3  Ampere    600 GB/s    12    50 GB/s     链路数翻倍
NVLink 4  Hopper    900 GB/s    18    50 GB/s     NVSwitch第三代，全连接
NVLink 5  Blackwell 1,800 GB/s  18    100 GB/s    NVL72，72GPU全互联域
NVLink 6  Rubin     3,600 GB/s  18    200 GB/s    NVL72/NVL144，光互联
```

### 5.3 NVSwitch：从机内到机柜的跨越

NVSwitch是NVLink的"交换机"。它的作用是让不直接相连的GPU之间也能通过NVLink通信，实现全互联拓扑。

```
NVSwitch演进：

第1代 (Volta):   18端口，支持8 GPU全互联（DGX-1/V100）
第2代 (Ampere):  改进带宽，8 GPU全互联（DGX A100）
第3代 (Hopper):  64端口，支持8 GPU全互联 + 跨节点NVLink Network
第4代 (Blackwell): 72端口，支持NVL72——一个机柜内72个GPU全互联
第5代 (Rubin):   72端口@200GB/s，支持NVL72/NVL144

NVL72架构（Blackwell/Rubin）：
┌─────────────────────────────────────────────┐
│                  NVL72 机柜                   │
│                                               │
│  ┌─────────┐ ┌─────────┐     ┌─────────┐    │
│  │GPU tray │ │GPU tray │ ... │GPU tray │    │  36个GPU tray
│  │(2 GPU)  │ │(2 GPU)  │     │(2 GPU)  │    │  共72个GPU
│  └────┬────┘ └────┬────┘     └────┬────┘    │
│       │           │               │          │
│  ┌────┴───────────┴───────────────┴────┐     │
│  │          NVSwitch tray 层            │     │  9个Switch tray
│  │     (每tray含多个NVSwitch芯片)        │     │  18个NVSwitch芯片
│  │     实现72 GPU Any-to-Any全互联       │     │
│  └─────────────────────────────────────┘     │
│                                               │
│  总带宽：130 TB/s (NVLink 5) / 260 TB/s (NVLink 6)│
│  → 超过整个互联网的总流量                       │
└─────────────────────────────────────────────┘
```

### 5.4 NVLink的协议层特性

NVLink不仅仅是"更快的线"。它在协议层提供了以太网/IB所不具备的能力：

**内存语义（Memory Semantics）**：GPU可以像访问本地显存一样访问NVLink对端GPU的显存（Load/Store操作），而不是消息传递语义。

**缓存一致性（Cache Coherence）**：从NVLink 2.0开始支持CPU-GPU之间的缓存一致性。NVLink-C2C进一步实现了CPU与GPU之间的高速coherent互联（Rubin平台上1.8 TB/s）。

**硬件原子操作（Atomic Operations）**：支持跨GPU的原子读-改-写操作，这对于同步原语至关重要。

### 5.5 NVLink Fusion：打开围墙花园

NVIDIA在2025年推出了NVLink Fusion，允许第三方CPU和XPU（定制加速器）通过NVLink-C2C接口接入NVLink域。这意味着非NVIDIA的芯片也可以加入NVLink的高速互联网络，是NVIDIA面向超大规模客户的生态开放策略。

### 7.2 NCCL深入解析

NCCL（NVIDIA Collective Communications Library）是大模型训练中最核心的通信库。理解NCCL就是理解GPU集群通信的关键。

**集合通信原语**：

```
AllReduce：所有节点贡献数据 → 聚合（如求和）→ 所有节点得到结果
  用途：数据并行的梯度同步（最频繁的操作）
  
AllGather：每个节点有一份数据 → 所有节点收集到完整数据
  用途：张量并行中收集分片结果
  
ReduceScatter：所有节点贡献数据 → 聚合 → 结果分片到各节点
  用途：ZeRO优化器状态分片

All-to-All：每个节点向每个其他节点发送不同数据
  用途：MoE专家并行的token dispatch/combine（最复杂的通信模式）
  
Broadcast：一个节点广播到所有节点
Reduce：所有节点聚合到一个节点
```

**NCCL的拓扑感知**：

NCCL在初始化时会自动探测系统的物理拓扑（哪些GPU通过NVLink相连、哪些通过PCIe、哪些需要跨节点），然后选择最优的通信算法和路径：

```
NCCL路径选择逻辑：

同一NVLink域内的GPU？
  → 使用NVLink直传（最快，亚微秒延迟）
  
同一节点但无NVLink？
  → 使用PCIe P2P或通过CPU中转
  
跨节点？
  → 检测IB/RoCE网卡
  → 使用GPUDirect RDMA（如果NIC和GPU在同一PCIe域）
  → 否则回退到CPU bounce buffer
  
NCCL算法选择：
  小消息量 → Tree算法（低延迟，O(log N)步）
  大消息量 → Ring算法（高带宽利用率，N步完成）
  支持SHARP → 使用CollNet（在网计算）
```

**NCCL通道（Channel）机制**：

NCCL会创建多个通信通道（Channel），每个通道独立执行一组集合操作。通过多通道并行，NCCL可以充分利用多条NVLink链路和多张网卡。

### 7.3 DeepEP：面向MoE的通信革新

DeepSeek开源的DeepEP通信库是MoE通信优化的标杆：

```
DeepEP核心创新：

1. 高吞吐All-to-All内核
   └─ GPU-initiated RDMA：由GPU SM直接发起RDMA操作
   └─ 减少CPU介入，降低调度开销
   
2. 通信-计算融合
   └─ MoE的dispatch和combine操作与专家计算流水线化
   └─ 不再是"先通信完再计算"
   
3. 低精度支持
   └─ FP8 dispatch减少通信量
   └─ 精度损失在MoE架构下可容忍
   
4. 两种模式：
   └─ 高吞吐模式（训练/Prefill）：最大化带宽利用
   └─ 低延迟模式（Decode）：最小化单次通信延迟
```

### 8.2 典型集群的混合并行通信架构

```
  ├── TP AllReduce：每个Transformer层 × 2次

  ├── SP AllGather/ReduceScatter

  └── EP跨节点All-to-All：MoE dispatch/combine

  TP ← 吃NVLink带宽（每层都通信，延迟决定一切）
  DP ← 吃IB/RoCE带宽（梯度大，但可overlap）
  EP ← 同时吃NVLink和IB（最复杂的通信模式）
═══════════════════════════════════════════════
```

### 8.3 通信优化技术

```
优化技术                    原理                         效果
──────────────────────────────────────────────────────────────
通信-计算Overlap            梯度产生后立即启动AllReduce     隐藏~70-90%通信时间
                           不等所有层计算完毕
                           
梯度压缩/量化通信            FP32梯度 → FP16/BF16/FP8     通信量降低2-4倍
                           TopK稀疏化                    
                           
分桶（Bucketing）           将小梯度张量合并为大桶          减少通信次数，提升带宽利用率
                           再执行AllReduce

异步通信                    DP梯度同步不阻塞前向计算        提升Pipeline效率
                           但需注意收敛性

SHARP在网计算               AllReduce在交换机内完成聚合     通信量减半（仅IB支持）

多轨并行（Multi-Rail）       单节点多张网卡同时通信          线性提升跨节点带宽
                           NCCL自动负载均衡               如8卡8网卡 = 8×400G

拓扑感知调度                 将TP组映射到NVLink域            避免跨域通信
                           将EP组映射到网络近邻             减少跳数
```

## 九、大模型推理中的通信需求

推理的通信特征与训练截然不同：

### 9.1 Prefill vs Decode：两种截然不同的通信模式

```
                Prefill阶段                  Decode阶段
──────────────────────────────────────────────────────────
计算特征        大batch矩阵运算              逐token自回归
                Compute-bound               Memory-bound
                
通信需求        类似训练的TP通信              极小通信量但极度延迟敏感
                AllReduce/AllGather          每生成一个token就要通信一次
                带宽敏感                      延迟敏感（直接影响TPOT）
                
NVLink价值      高（大量数据搬运）            极高（1μs vs 5μs差距
                                            被token生成速度放大）
                                            
典型规模        单请求可能用满TP组            单请求的TP通信量极小
                                            但延迟必须极低
```

### 9.2 Disaggregated Inference（分离式推理）

这是2025-2026年推理架构的核心趋势，它引入了全新的通信需求：

```
传统推理：Prefill和Decode在同一组GPU上
分离式推理：Prefill和Decode在不同GPU组上

┌──────────────┐         KV Cache 传输         ┌──────────────┐
│  Prefill GPU  │  ─────────────────────────→  │  Decode GPU   │
│  组（算力型） │  通过RDMA/NVLink传输KV Cache   │  组（延迟型） │
│  高吞吐优化   │                               │  低延迟优化   │
└──────────────┘                               └──────────────┘

新的通信挑战：
  1. KV Cache尺寸可能很大（长序列 × 多层 × 多头 × 隐藏维度）
     例如：128K context × 96层 × 128头 × 128维 × FP16 ≈ 数百GB
  2. TTFT(首Token延迟)要求 < 500ms
     如果Prefill耗时200ms，KV传输只剩300ms
  3. 需要高带宽、低延迟的节点间传输 → RDMA / NVLink Network

vLLM、TensorRT-LLM等推理框架已支持此架构
```

### 9.3 MoE推理中的EP通信

```
DeepSeek-V3推理通信模式：

每生成一个token：
  1. Router计算 → 确定激活哪些专家
  2. All-to-All dispatch → token embedding发送到专家所在GPU
  3. 专家计算
  4. All-to-All combine → 结果回收到原始GPU
  
Decode阶段每个token只有极小的数据量但需要完整的A2A
→ 通信延迟直接叠加到每个token的生成时间
→ 这就是DeepEP"低延迟模式"存在的原因
```

## 十一、未来趋势

### 11.1 光互联（CPO/硅光）

当前电信号互联面临功耗和距离瓶颈。NVIDIA Rubin平台已开始在NVLink switch tray中引入光学互联。CPO（Co-Packaged Optics）将光引擎直接与交换芯片封装在一起，预计2030年在AI数据中心光通信模块中渗透率达35%。这将从根本上改变互联的功耗和距离约束。

### 11.2 机柜级计算成为新范式

从NVL72到NVL144，从灵衢超节点，趋势非常清晰：**机柜即计算机**。一个机柜内的72-144个GPU通过NVLink/灵衢UB形成一个巨大的"虚拟GPU"，共享统一的内存地址空间。机间网络（IB/RoCE）退化为机柜间的互联。

### 11.3 协议竞争格局

```
三极格局：

NVIDIA垂直整合：NVLink + ConnectX + Spectrum + SHARP + NCCL
  → 从芯片到软件全栈掌控
  → 性能最优，但供应商锁定

以太网开放阵营：RoCE + Ultra Ethernet + 多厂商
  → Meta、Google、Microsoft推动
  → 成本低、生态广、性能追赶中

华为开放超节点：灵衢UB + HCCL + 昇腾 + openEuler
  → 协议归一、规范开放
  → 中国AI基础设施的核心支撑
  → 正在构建独立生态
```
