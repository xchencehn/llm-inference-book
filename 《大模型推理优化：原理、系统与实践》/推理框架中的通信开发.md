
## 一、先建立一个核心认知：推理通信和训练通信是两个世界

你在上一轮了解了训练时通信的样子。推理的开发者面对的是一个完全不同的战场：

```
训练时的通信：                          推理时的通信：
                                       
大batch (数千个样本)                    可能只有1个请求在decode
每步通信量大 (MB~GB级)                  每步通信量极小 (KB~几十KB级)
吞吐优先 (带宽利用率)                   延迟优先 (每多1μs都在损失用户体验)
通信可以和计算overlap                   decode阶段计算极快，通信无处可藏
梯度同步有规律                          MoE的token路由不规则
一次训练跑几天                          一个请求几百毫秒要出结果
NCCL够用了                             NCCL对小消息太慢，必须自己搞
```

用一句话概括差异：训练怕的是带宽不够，推理怕的是延迟太高。

## 二、推理框架面对的三类通信问题

vLLM 和 SGLang 的开发者需要处理三类完全不同的通信场景，每类的解决方式都不一样：

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                    │
│  场景1：张量并行（TP）的AllReduce                                    │
│  └─ 每生成一个token都要做，延迟直接叠加到用户感知                      │
│  └─ 数据量小（几十KB），NCCL反而不是最优                              │
│  └─ vLLM的解法：自研Custom AllReduce CUDA Kernel                    │
│                                                                    │
│  场景2：MoE专家并行（EP）的All-to-All                                │
│  └─ DeepSeek-V3等MoE模型推理的核心瓶颈                               │
│  └─ 不规则通信模式，每个token去不同专家                                │
│  └─ SGLang的解法：集成DeepEP通信库                                   │
│                                                                    │
│  场景3：Prefill-Decode分离的KV Cache传输                             │
│  └─ 全新的通信类型，不是集合通信，而是大块数据搬运                      │
│  └─ GB级数据要在不同GPU节点间快速搬运                                  │
│  └─ vLLM的解法：NIXL（NVIDIA Inference Xfer Library）               │
│                                                                    │
└──────────────────────────────────────────────────────────────────┘
```

我逐个展开。

## 三、场景1：TP的AllReduce——为什么vLLM要自己写通信kernel

### 3.1 问题在哪

你已经知道，张量并行的Transformer层里每一层都要做AllReduce。在推理的decode阶段，每生成一个token就要走完所有层，意味着AllReduce被调用**几十到几百次**。

关键在于数据量。训练时，大batch意味着AllReduce的tensor很大（MB级），NCCL的Ring/Tree算法在大消息上效率极高。但decode阶段，batch可能只有1个token，AllReduce的tensor只有：

```
hidden_size × dtype_size = 8192 × 2 bytes (BF16) = 16 KB
```

16KB。NCCL对这种小消息有一个致命问题——**启动开销（launch overhead）远大于实际数据传输时间**。NCCL需要经过一长串调度逻辑：选择算法、分配通道、启动kernel、同步屏障……这套流程本身就要好几微秒，而实际在NVLink上传16KB数据不到1微秒。

```
NCCL的AllReduce延迟 ≈ 启动开销(~8μs) + 传输时间(~0.5μs) ≈ 8.5μs

如果一个96层模型每层2次AllReduce，一个token要：
96 × 2 × 8.5μs = 1,632μs ≈ 1.6ms 花在通信上

而decode的计算本身可能只要 ~2ms
→ 通信占了总延迟的 ~45%！
```

### 3.2 vLLM的Custom AllReduce：直接手写CUDA kernel

vLLM的开发者做了一件看起来很疯狂的事——**绕过NCCL，自己写了一个AllReduce的CUDA kernel**。这个kernel直接利用NVLink的硬件特性，用最原始、最轻量的方式完成通信。

核心思路：

```
NCCL的做法（通用但重）：
  Ring/Tree算法 → 多步通信 → 每步kernel launch → 多次同步

vLLM Custom AllReduce的做法（专用但轻）：
  利用NVLink提供的GPU间直接内存访问能力
  所有GPU在一个kernel里直接读取其他GPU的显存
  一次kernel launch完成所有事情
```

具体实现分为两个阶段：

```c++
// 伪代码：vLLM custom_all_reduce.cuh 的核心逻辑

// 前提：所有GPU的显存通过NVLink互相可见（P2P mapped）
// vLLM在初始化时用 cudaIpcOpenMemHandle 把每个GPU的buffer映射到所有GPU

__global__ void custom_allreduce_kernel(
    float** all_gpu_buffers,  // 指向所有GPU的buffer的指针数组
    float* output,
    int N, int world_size
) {
    int tid = threadIdx.x + blockIdx.x * blockDim.x;
    
    if (tid < N) {
        float sum = 0.0f;
        // 直接从每个GPU的显存里读数据（通过NVLink）
        for (int r = 0; r < world_size; r++) {
            sum += all_gpu_buffers[r][tid];  // ← NVLink load!
        }
        output[tid] = sum;
    }
}

// 实际上分两个阶段：
// Stage 1 (one-shot, 适用于小消息 < ~256KB):
//   每个GPU直接读取所有其他GPU的数据并求和
//   一次kernel launch搞定，无需多轮通信
//
// Stage 2 (two-shot, 适用于较大消息):
//   先ReduceScatter：每个GPU负责一部分数据的聚合
//   再AllGather：把聚合结果广播给所有人
//   两次kernel launch，每次通信量更小
```

这个方案为什么对推理特别有效：

```
vLLM Custom AllReduce 的优势：
                                          
1. 一次kernel launch搞定（NCCL可能需要多次）        
   → 消除了多次launch的累积开销                     
                                                  
2. 无需NCCL的复杂调度逻辑                           
   → 启动延迟从~8μs降到~2-3μs                      
                                                  
3. 与CUDA Graph兼容                                
   → decode阶段可以把整个计算图（含通信）capture成Graph
   → 后续每步直接replay，几乎零开销                  
                                                  
4. 在NVLink全互联拓扑下最优                         
   → 专为DGX/HGX的8卡NVLink拓扑设计                
   → 不需要Ring/Tree这种通用拓扑算法                 
```

### 3.3 这在代码里是怎么体现的

如果你去读vLLM的模型代码，你会看到它的Transformer层和Megatron-LM长得很像，但通信调用走了不同的路径：

```python
# vLLM中一个典型的RowParallelLinear层的forward
# 文件: vllm/model_executor/layers/linear.py (简化)

class RowParallelLinear(nn.Module):
    def forward(self, input_):
        # 本地矩阵乘法（和训练一模一样）
        output = F.linear(input_, self.weight)
        
        # ★ 通信部分：不直接调NCCL，而是走vLLM的通信抽象层 ★
        if self.reduce_results and self.tp_size > 1:
            output = tensor_model_parallel_all_reduce(output)
            #         ↑ 这个函数内部会判断：
            #           → 数据量小 + NVLink全互联？ → Custom AllReduce kernel
            #           → 数据量大 或 非NVLink？   → 回退到NCCL
        
        return output


# vllm/distributed/parallel_state.py 里的路由逻辑(简化)
def tensor_model_parallel_all_reduce(tensor):
    group = get_tp_group()
    
    if group.use_custom_allreduce:
        # 走自研的Custom AllReduce
        return group.ca_comm.custom_all_reduce(tensor)
    else:
        # 回退到标准NCCL
        torch.distributed.all_reduce(tensor, group=group.device_group)
    
    return tensor
```

你看到了关键点：**vLLM的模型层代码里确实写了通信调用**（`tensor_model_parallel_all_reduce`），但这些代码不是你作为算法工程师写的——它们被vLLM开发者内置在了模型层的定义里。当你用vLLM加载一个HuggingFace模型时，vLLM会自动把模型的`nn.Linear`替换成它自己的`ColumnParallelLinear`/`RowParallelLinear`，通信就自动插进去了。

### 3.4 CUDA Graph如何把通信也"录"进去

decode阶段的一个重要优化是CUDA Graph：把多个kernel的launch开销降到一次GPU submit。但NCCL的通信kernel很难被录入CUDA Graph（因为NCCL内部有复杂的状态管理）。vLLM的Custom AllReduce天然兼容CUDA Graph：

```
没有CUDA Graph：
  [launch kernel 1] → [launch allreduce] → [launch kernel 2] → [launch allreduce] → ...
  每次launch: ~3-5μs CPU开销
  96层 × 多个kernel = 巨大的累积launch开销

有CUDA Graph（含Custom AllReduce）：
  [一次性replay整个Graph]
  → kernel1, allreduce, kernel2, allreduce, ... 全在GPU上连续执行
  → CPU只提交一次，GPU自己跑完所有东西
  → 省掉了所有中间的launch开销
```

## 四、场景2：MoE专家并行——SGLang如何集成DeepEP

### 4.1 问题在哪

当你部署DeepSeek-V3这样的MoE模型进行推理时，专家分布在不同GPU上。每个token需要被路由到正确的专家所在的GPU——这是一个All-to-All通信。

```
假设8个GPU，256个专家，EP=8（每GPU 32个专家）
一个batch里100个token，每个token激活8个专家

Dispatch阶段：
  GPU0上的token_1需要专家17（在GPU0）、专家55（在GPU1）、专家200（在GPU6）...
  GPU0上的token_2需要专家3（在GPU0）、专家77（在GPU2）、专家150（在GPU4）...
  GPU1上的token_3需要专家20（在GPU0）、专家99（在GPU3）...
  ...
  
→ 每个GPU要向每个其他GPU发送不同数量的不同token
→ 完全不规则的通信模式
→ NCCL的All-to-All对此效率不高
```

### 4.2 SGLang的方案：集成DeepEP

SGLang（以及vLLM）集成了DeepSeek开源的DeepEP通信库来处理这个问题。DeepEP的核心创新我们展开看：

```python
# SGLang中MoE层的推理通信 (简化概念)
# 实际代码在 sglang/srt/layers/moe/ 目录下

class MoELayer:
    def forward(self, hidden_states):
        # 1. 路由计算（本地）
        router_logits = self.gate(hidden_states)
        topk_weights, topk_ids = torch.topk(router_logits, k=self.top_k)
        
        # 2. ★ DeepEP dispatch: All-to-All 发送token到专家所在GPU ★
        if self.use_deepep:
            # DeepEP提供两种模式，框架根据当前阶段自动选择
            if is_prefill_phase:
                # 高吞吐模式：大batch，优化带宽利用率
                dispatched = self.deepep_comm.dispatch(
                    hidden_states, topk_ids, 
                    mode="high_throughput"
                )
            else:
                # 低延迟模式：decode阶段，单/少量token
                dispatched = self.deepep_comm.dispatch(
                    hidden_states, topk_ids,
                    mode="low_latency"
                )
        else:
            # 回退到标准NCCL all_to_all
            dispatched = nccl_all_to_all(hidden_states, topk_ids)
        
        # 3. 专家计算（本地）
        expert_output = self.experts(dispatched)
        
        # 4. ★ DeepEP combine: All-to-All 把结果发回原始GPU ★
        combined = self.deepep_comm.combine(expert_output)
        
        return combined
```

DeepEP的两种模式针对推理的两个阶段做了截然不同的优化：

```
DeepEP高吞吐模式（Prefill阶段）：
┌──────────────────────────────────────────────────────┐
│  大量token需要dispatch                                │
│  → 攒够一批token再一次性发出                           │
│  → 利用NVLink/IB带宽，追求吞吐量最大化                 │
│  → 使用GPU SM（流处理器）直接发起RDMA操作              │
│     不经过CPU → 省去CPU调度开销                        │
│  → 支持FP8量化dispatch → 通信量减半                   │
│  → 通信和专家计算pipeline overlap                      │
└──────────────────────────────────────────────────────┘

DeepEP低延迟模式（Decode阶段）：
┌──────────────────────────────────────────────────────┐
│  每次只有极少量token（甚至1个）                         │
│  → 不攒batch，立即发送                                │
│  → 一切以最小延迟为目标                                │
│  → 专门优化的小消息传输kernel                          │
│  → 使用NVLink的LL（Low Latency）协议                  │
│  → 减少同步点                                         │
│  → Decode阶段每token的EP延迟从~50μs降到~10-20μs      │
└──────────────────────────────────────────────────────┘
```

### 4.3 节点内EP vs 跨节点EP

一个很实际的问题：如果专家并行只在8卡NVLink域内，通信走NVLink，延迟低。但DeepSeek-V3有256个专家，8卡放不下，必须跨节点：

```
节点内EP（8卡NVLink）：
  All-to-All走NVLink
  延迟：~几μs
  带宽：1.8 TB/s（NVLink 5）
  → 体验还可以

跨节点EP（64卡，跨8台机器）：
  All-to-All走IB/RoCE网络
  延迟：~几十μs
  带宽：400-800 Gbps/端口
  → Decode阶段每个token的延迟显著增加
  
SGLang的优化策略：
  1. 尽量让TP组在NVLink内，EP组也优先NVLink内
  2. DeepSeek-V3推理典型配置：TP=1, EP=8（全在NVLink内）
     或 TP=1, EP=64（跨节点，用DeepEP低延迟模式）
  3. 专家负载均衡(EPLB)：动态复制热门专家到多个GPU
     → 减少跨节点的All-to-All流量
```

## 五、场景3：PD分离的KV Cache传输——一种全新的通信类型

### 5.1 这和前两种完全不同

TP的AllReduce和EP的All-to-All都是**集合通信**——多个GPU之间交换数据。但Prefill-Decode分离引入了一种新的通信模式：**点对点的大块数据搬运**。

```
Prefill GPU (计算完KV Cache)  ──→  Decode GPU (需要KV Cache来生成token)

KV Cache的大小：
  layers × 2(K和V) × num_heads × head_dim × seq_len × dtype_size
  例如：96层 × 2 × 128头 × 128维 × 4096 seq_len × 2 bytes(FP16)
     = 96 × 2 × 128 × 128 × 4096 × 2 ≈ 24 GB

24GB数据要在Prefill完成后立刻传给Decode节点
用户在等首个token（TTFT），每一毫秒都重要
```

### 5.2 vLLM的NIXL方案

vLLM集成了NVIDIA的NIXL（NVIDIA Inference Xfer Library）来做这件事。NIXL本质上就是一个优化过的RDMA数据搬运引擎：

```python
# vLLM PD分离的通信架构 (概念图)

# Prefill实例：
class PrefillInstance:
    def process_request(self, prompt):
        # 1. 正常执行prefill计算
        kv_cache = self.model.prefill(prompt)
        
        # 2. ★ 通过NIXL把KV Cache推送/拉取到Decode实例 ★
        # 当前vLLM默认是"pull模式"：Decode端主动来拉取
        self.nixl_agent.register_kv_cache(request_id, kv_cache)
        # NIXL底层做的事：
        # → 把KV Cache所在的GPU显存注册为RDMA Memory Region
        # → 通知Decode端：这块数据准备好了
        
# Decode实例：
class DecodeInstance:
    def start_decode(self, request_id):
        # 1. ★ 通过NIXL拉取KV Cache ★
        kv_cache = self.nixl_agent.pull_kv_cache(
            source=prefill_node_addr,
            request_id=request_id
        )
        # NIXL底层做的事：
        # → 发起RDMA READ操作
        # → Prefill端的HCA直接从GPU显存DMA读出数据
        # → 通过IB/RoCE网络传输
        # → Decode端的HCA直接写入本地GPU显存
        # → 全程CPU不参与数据拷贝（零拷贝）
        # → 异步执行，可以和其他操作overlap
        
        # 2. 用收到的KV Cache开始自回归decode
        for token in self.model.decode(kv_cache):
            yield token
```

NIXL相比你自己用 `torch.distributed.send/recv` 的优势：

```
torch.distributed.send/recv:
  → 经过NCCL → 经过通用的通信调度 → 同步等待
  → 不支持CUDA Graph
  → 对大块非连续内存传输不够优化

NIXL:
  → 直接操作RDMA verbs层
  → 完全异步，支持non-blocking传输
  → 针对KV Cache的block结构做了零拷贝优化
     (vLLM的KV Cache是分block管理的，NIXL可以直接传输scattered blocks)
  → 支持多轨并行（同时用多张网卡传输）
  → 支持传输过程中Decode已经开始生成（pipeline）
```

## 七、对比总结：训练框架 vs 推理框架处理通信的差异

```
╔══════════════════════════════════════════════════════════════════════════════╗
║                       训练框架                    推理框架                    ║
║  维度                 (PyTorch DDP/Megatron)      (vLLM/SGLang)             ║
╠══════════════════════════════════════════════════════════════════════════════╣
║                                                                              ║
║  有没有backward      有，通信的大头在backward      没有backward！              ║
║                      (DDP梯度同步)                所有通信都在forward          ║
║                                                                              ║
║  通信频率            每个训练step一次              每生成一个token就通信        ║
║                      (数秒一次)                   (几毫秒一次)                ║
║                                                                              ║
║  通信数据量          大(MB~GB)                    小(KB~几十KB per token)      ║
║                                                   大(GB级 for KV传输)         ║
║                                                                              ║
║  核心瓶颈            带宽                          延迟                        ║
║                                                                              ║
║  通信库              主要用NCCL                    NCCL + 自研kernel + DeepEP  ║
║                      NCCL够用                     + NIXL                     ║
║                                                   NCCL对小消息不够快           ║
║                                                                              ║
║  CUDA Graph          训练基本不用                   Decode阶段重度依赖          ║
║                      (动态shape)                   (固定shape可capture)       ║
║                                                   通信kernel必须兼容Graph      ║
║                                                                              ║
║  通信-计算overlap    DDP梯度和计算overlap           Decode阶段几乎没法overlap   ║
║                      是主要优化手段                 因为计算本身太快             ║
║                                                                              ║
║  新增通信类型         无                           KV Cache跨节点搬运(PD分离)   ║
║                                                                              ║
║  通信对算法工程师      完全透明                     完全透明                     ║
║  的可见性             (DDP包一下就行)               (启动时 --tp 8 就行)        ║
║                                                                              ║
║  通信对框架开发者      调用NCCL API                 自己写CUDA kernel           ║
```

## 八、最后一个你可能会问的问题：那我作为使用者，需要做什么？

答案是：几乎什么都不用做，但需要**理解配置的含义**。

```bash
# vLLM启动命令示例
python -m vllm.entrypoints.openai.api_server \
    --model deepseek-ai/DeepSeek-V3 \
    --tensor-parallel-size 8 \        # ← 你选了TP=8，框架自动处理192次AllReduce
    --pipeline-parallel-size 2 \      # ← 你选了PP=2，框架自动处理P2P通信
    --enable-expert-parallel \        # ← 开启EP，框架自动处理All-to-All
    --disable-custom-all-reduce \     # ← 你可以关掉自研kernel回退NCCL（调试用）
    --kv-transfer-config '...'        # ← PD分离时配置KV传输方式

# SGLang启动命令示例
python -m sglang.launch_server \
    --model deepseek-ai/DeepSeek-V3 \
    --tp 8 --ep 64 \                 # ← TP和EP配置
    --enable-deepep \                 # ← 启用DeepEP通信库
    --deepep-mode low_latency         # ← 选择DeepEP的模式
```

你要做的选择本质上就是这些数字——`tp`、`ep`、`pp`。但这些数字背后涉及的通信模式、硬件路径、性能影响，就是前面全部内容所讲的东西。知道NVLink域只有8卡，你就理解了为什么`tp`最好是8或者8的因子。知道跨节点EP要走IB/RoCE，你就理解了为什么DeepSeek-V3推理在没有高速网络时EP性能会崩。知道Decode阶段通信延迟敏感，你就理解了为什么vLLM要费力自己写Custom AllReduce而不直接用NCCL。

一切都连上了。
