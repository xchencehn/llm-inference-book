

TensorRT-LLM 的核心哲学是**编译时优化**（Compile-time Optimization）：不是在运行时用通用算子逐个执行模型的每一步，而是在部署前对整个模型进行分析，将多个算子融合、针对目标硬件的具体特性（SM 数量、寄存器文件大小、共享内存容量、Tensor Core 类型等）生成高度特化的 Kernel，然后在推理时直接执行这些预编译好的"引擎"。这条技术路线已经演化为一个庞大的体系，玩家横跨学术界和工业界。

## 一、技术分层：从算子融合到 MegaKernel

要理解这个方向的全貌，需要先理清它的**技术谱系**——按融合粒度从小到大排列。

### 第一层：手写定制 Kernel + Kernel 库

这是最基础也最"暴力"的方式——由专家针对特定算子手写高度优化的 CUDA/CUTLASS Kernel。

**FlashInfer**（MLSys 2025 Best Paper）是这个层次的典范。它由 CMU 的 Zihao Ye 等人开发，是一个专为 LLM 推理设计的 GPU Kernel 库，核心创新在于：用统一的 Block-Sparse 格式表示各种异构 KV Cache 布局（Ragged Tensor、Paged KV Cache、Cascade 等），并通过 JIT 编译的 CUDA/CUTLASS 模板让用户可以**自定义注意力变体**（如 MLA、Sliding Window、Cross-Attention 等），无需为每种变体单独写 Kernel。FlashInfer 已被 SGLang 和 vLLM 作为默认 Attention 后端集成，NVIDIA 也官方将其纳入推荐方案。

**CUTLASS**（NVIDIA）是更底层的 CUDA C++ 模板库，提供高度可组合的 GEMM/Attention 原语。TensorRT-LLM 和 FlashInfer 底层大量使用 CUTLASS。CUTLASS 本身不直接面向推理引擎用户，但它为上层编译器提供了构建高性能 Kernel 的基石。

**ThunderKittens**（Stanford HazyResearch）是另一个值得关注的 Kernel DSL，它将 GPU 编程抽象到"warp 级别的小矩阵操作"上，让编写高效 Attention Kernel 变得更简单。Stanford 团队用它实现了 Llama-1B 的 MegaKernel 原型（后面会详述）。

**Liger Kernel**（LinkedIn）则是面向 Triton 的 Kernel 集合，通过将 RMSNorm、SwiGLU、CrossEntropy 等常用算子用 Triton 重写并融合，在 LLM 训练中实现了 20% 的吞吐提升和 60% 的显存节约。虽然最初面向训练，其中的 Kernel 融合思想同样适用于推理。

### 第二层：JIT 编译 + 自动算子融合

这一层的核心思路是：不需要手写每个 Kernel，而是通过编译器**自动发现并融合**可融合的算子序列，JIT（Just-In-Time）生成优化后的 Kernel。

**torch.compile（PyTorch/TorchInductor）** 是当前生态中最重要的 JIT 编译方案。它的工作流程分为两步：前端 TorchDynamo 通过 Python 字节码解释器将 PyTorch 代码捕获为 `torch.fx` 计算图；后端 TorchInductor 对图执行 pointwise/reduction 融合、matmul 后端选择（cuBLAS/Triton/CUTLASS）、Prologue/Epilogue 融合、CUDA Graph 自动化等优化，最终生成 Triton 或 C++ Kernel。

torch.compile 在推理引擎中的集成情况非常值得详述：

**vLLM V1** 已将 torch.compile **默认开启**。根据 vLLM 团队 2025 年 8 月的技术博客，他们在标准的 TorchInductor 优化基础上叠加了大量**自定义编译器 Pass**，包括：RMSNorm + FP8 量化融合（最高 15% 提升）、SiLU-Mul + FP8/FP4 量化融合（约 8% 提升）、Attention + FP8 量化融合（约 7% 提升）、AllReduce + RMSNorm 融合（约 15% 提升）、Sequence Parallelism + Async TP（约 10% 提升）。这些 Pass 的精妙之处在于：模型代码保持干净的模块化写法，融合优化全部在编译期的图变换中完成——模型作者负责正确性，编译器负责性能，实现了**关注点分离**。vLLM 还支持 Piecewise CUDA Graphs（自动将图分为 CUDA Graph 兼容/不兼容的部分分别执行）和编译缓存（首次编译后缓存 artifact，后续启动直接加载）。

**SGLang** 同样深度集成了 torch.compile，并在 2025 年 9 月开启了一个**统一 Kernel 融合 RFC**，目标是在 SGLang 中建立系统化的融合框架。其与 torch.compile 的集成面临的一个挑战是**启动时间**——编译开销在自动伸缩场景中是个痛点。

**OpenAI Triton** 在这个体系中扮演了关键角色：它既是 torch.compile/TorchInductor 的默认代码生成后端之一，也是独立的 GPU 编程语言。开发者用类 Python 的语法写 Kernel，Triton 编译器自动处理 Tiling、共享内存管理、向量化等底层优化。NVIDIA 在 Blackwell 架构上进一步增强了 Triton 支持，引入了 **CUDA Tile IR 后端**（2026 年 1 月发布），让 Triton Kernel 可以直接编译到 NVIDIA 最新的 Tile IR 层级，获得与 CUTLASS 接近的性能。

### 第三层：AOT 编译——静态全图优化

与 JIT 编译的"运行时编译"不同，AOT（Ahead-Of-Time）编译在部署前将整个模型编译为硬件特化的执行引擎。

**TensorRT-LLM（NVIDIA）** 是这个方向的标杆。它将 LLM 的完整计算图输入 TensorRT 优化器，执行层融合、精度校准（FP8/INT8/INT4）、Kernel 自动调优（Auto-Tuning）、内存布局优化等，生成一个**序列化的 engine 文件**。这个 engine 文件针对特定的模型架构、序列长度范围和 GPU 型号高度特化。其优势是在目标硬件上的单请求延迟极低（尤其在 NVIDIA GPU 上通常是性能天花板），劣势是 engine build 过程慢（分钟到小时级别）、模型变更需要重新编译、且完全绑定 NVIDIA 硬件。据 NVIDIA 报告，TensorRT-LLM 在支持的配置上比 llama.cpp 快 30-70%。

**XLA（Google）** 是 Google 为 TPU 和 GPU 构建的 AOT 编译器，JAX 框架的默认编译后端。XLA 对整个计算图进行静态分析，执行算子融合、缓冲区分配优化、并行度调度等，生成高效的 HLO（High Level Optimizer）IR 并最终编译到 TPU/GPU 代码。Google 的 **JetStream** 和 **MaxText** 推理框架都建立在 JAX+XLA 之上，为 TPU 上的 Gemini 等模型服务。2025 年末，vLLM 也引入了 TPU 后端，通过 Torchax 桥接支持 JAX 和 XLA 编译。XLA 在 TPU 上的优势尤为明显——TPU 的架构设计本身就是围绕 XLA 的静态图编译模型构建的。

**Apache TVM / MLC LLM（陈天奇团队）** 走的是**跨硬件万能编译**路线。TVM Unity（TVM 的最新版本）将模型编译到一个中间表示（Relax IR），然后通过自动化的搜索和调优为目标硬件生成最优 Kernel。**MLC LLM** 是建立在 TVM Unity 之上的 LLM 专用部署引擎，其核心价值主张是**一次编写、到处运行**——同一个模型可以编译部署到 NVIDIA GPU（CUDA）、Apple Silicon（Metal）、AMD GPU（Vulkan/ROCm）、Web 浏览器（WebGPU）、Android/iOS 等所有平台，且每个平台都获得经过编译器优化的高效 Kernel。MLC LLM 还推出了 **WebLLM**——完全在浏览器中运行 LLM 推理的方案。MLC LLM 的跨平台能力是其他框架难以匹敌的，但在单一 NVIDIA GPU 上的极致性能通常略逊于 TensorRT-LLM 或 vLLM+torch.compile。

**Modular MAX/Mojo（Chris Lattner 团队）** 是 LLVM 之父 Chris Lattner 创办的 Modular 公司的作品。MAX 是一个 AI 推理引擎，其底层使用 Mojo 语言（一种兼容 Python 语法但编译到原生代码的系统级语言）编写核心算子。MAX 的编译器对模型进行全图优化和 Kernel 融合，号称在 CPU 上比 PyTorch 快 10%-650%，在 GPU 上也有显著加速。Modular 的定位类似"下一代 TensorRT"但更开放——不绑定 NVIDIA，支持多种硬件后端。

### 第四层：MegaKernel——将整个模型编译为一个 Kernel

这是编译式优化的**终极形态**：不是融合几个相邻算子，而是将 LLM 的**整个前向传播**（甚至包括跨 GPU 通信）编译为一个**单一 GPU Kernel**。这彻底消除了所有 Kernel Launch 开销、中间结果的 HBM 读写，以及 CPU-GPU 同步延迟。

**Mirage Persistent Kernel（MPK，CMU/Zhihao Jia 团队）** 是这个方向最成熟的学术工作，2025 年 12 月发表。MPK 的编译器将 LLM 模型从 PyTorch 层级自动降低（lower）为一个 SM 级别的任务图，然后生成一个持久化 MegaKernel——所有 SM 在整个推理过程中持续运行同一个 Kernel，通过 Kernel 内部的并行运行时调度不同的计算任务（GEMM、Attention、LayerNorm 等）。MPK 的关键优势在于：消除 Kernel Launch 开销（对 Decode 阶段尤为关键，因为每步计算量小但 Launch 频繁），实现跨算子的数据流水线（一个算子的输出直接通过共享内存传给下一个算子，避免写回 HBM），以及将**跨 GPU 通信融入 Kernel 内部**（NCCL 通信也变成 Kernel 内的操作）。vLLM 已经在 RFC 中提出集成 MPK 的计划。

**Stanford HazyResearch 的 “Look Ma, No Bubbles”**（2025 年 5 月）是另一个 MegaKernel 实践——用 ThunderKittens DSL 将 Llama-1B 的完整前向传播手写为单个 MegaKernel，消除了所有 Kernel 边界的"气泡"。虽然是手写而非编译器自动生成，但验证了 MegaKernel 理念的可行性和性能潜力。

**Luminal（Joe Fioti）** 是一个用 Rust 编写的推理编译器，其 2026 年 1 月的博客详细阐述了将模型编译为 MegaKernel 的技术路径。Luminal 通过计算图优化将整个模型融合为尽可能少的 Kernel（理想情况下是一个），目标是成为"任何模型在任何设备上最快的 ML 框架"。

## 三、这个方向的关键趋势和现状

**趋势一：编译式优化正在成为推理引擎的标配**。vLLM V1 和 SGLang 都已将 torch.compile 作为默认组件，这意味着"手写 Kernel"不再是唯一的优化路径，编译器自动生成的 Kernel 正在逐步逼近甚至超越手写 Kernel 的性能。vLLM 团队的规划是将优化级别体系化为 `-O0`（纯 eager，最快启动）到 `-O3`（最长编译、最高性能），让用户按需权衡。

**趋势二：MegaKernel 是性能的下一个前沿**。当前主流方案中，即使经过 torch.compile 的融合和 CUDA Graph 的优化，模型前向传播仍然由数十到上百个 Kernel 组成，Kernel 之间的边界引入了 HBM 读写和 Launch 开销。MegaKernel 将这一切压缩为一个 Kernel 调用，理论上是 Decode 延迟优化的终极方案。MPK 和 Luminal 都在朝这个方向推进，vLLM 已将 MPK 集成列入路线图。

**趋势三：LLM 自己开始写 Kernel**。2025 年下半年出现了一个有趣的方向——用 LLM Agent 自动生成和优化 CUDA/Triton Kernel。NVIDIA 的 DeepSeek-R1 实验、字节跳动的 CUDA Agent（用 RL 训练 LLM 生成 Kernel）、PyTorch 团队的 KernelFalcon（Deep Agent 架构自动分解和委派 Kernel 生成任务），都在 KernelBench 等基准上展示了超越 torch.compile 自动生成 Kernel 的能力。这可能意味着未来的编译器后端不再是固定的规则引擎，而是一个由 LLM 驱动的自适应代码生成系统。

**趋势四：编译式优化正在扩展到通信层**。传统的 Kernel 融合局限于单 GPU 上的计算算子，而分布式推理中跨 GPU 的 AllReduce/AllGather 通信往往是更大的瓶颈。vLLM 的 Async TP Pass 将通信操作重构为 ReduceScatter + AllGather 并与 GEMM 融合，MPK 更是将 NCCL 通信直接嵌入 MegaKernel 内部。这标志着编译式优化从"单卡计算优化"扩展到"分布式计算+通信联合优化"。

**趋势五：编译与手写 Kernel 的混合是现实中的最优策略**。尽管编译器在不断进步，vLLM 团队坦率承认：涉及 custom ops（如特定的 Attention 变体、sub-byte 量化、集合通信）时仍然需要自定义的编译器 Pass 来进行融合，编译器无法自动处理所有情况。FlashInfer 这类手写 Kernel 库在 Attention 这一最关键的算子上仍然不可替代。现实中的最高性能方案是：**FlashInfer/CUTLASS 处理 Attention 和 GEMM 这些核心热点，torch.compile 自动融合其余的 pointwise/reduction 算子，自定义 Pass 处理跨算子的特殊融合模式**。

## 四、总结

如果把"为每个模型和硬件生成定制化 Kernel"这条技术路线比喻为一个光谱：

- **左端**是完全手写——FlashInfer、CUTLASS、ThunderKittens，性能最高但开发成本极高，每个模型/硬件组合都需要专家投入。
- **中间**是编译器自动化——torch.compile、XLA、TVM、Modular MAX，通过编译器分析自动融合和优化，开发成本低但可能遗漏某些优化机会。
- **右端**是 MegaKernel 编译器——MPK、Luminal，将自动化推到极致，但目前还在成熟中。
- **新维度**是 LLM 驱动的 Kernel 生成——CUDA Agent、KernelFalcon，用 AI 来写 AI 的 Kernel，是最前沿但也最不成熟的方向。

整个行业正在从左端向右端演进，同时各层次的技术在叠加使用。TensorRT-LLM 作为先行者建立了这个方向的基本范式，但它的封闭性和 NVIDIA 绑定正在被 torch.compile 生态的开放性和跨平台性挑战。未来 1-2 年，**编译式优化 + MegaKernel + AI 辅助 Kernel 生成**三者的融合，很可能定义下一代推理引擎的性能基准。
