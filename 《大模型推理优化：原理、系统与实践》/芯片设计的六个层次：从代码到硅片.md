

```
       抽象程度：高 ───────────────────────────────────→ 低
与物理硬件的距离：远 ────────────────────────────────────→ 近

  ISA指令集 → API/SDK → 白皮书 → 微架构 → RTL源码 → 物理版图
  (PTX/SASS)  (CUDA)   (规格数据) (NoC设计) (Verilog) (GDSII)
  
  "能做什么"  "怎么调用" "能做多快" "怎么连的" "怎么造的" "长什么样"
  
  你可以理解为：
  菜谱       → 厨房操作手册 → 餐厅广告 → 厨房设计图 → 施工图纸 → 实际建筑图
```

## 一、ISA指令集（PTX / SASS）


ISA（Instruction Set Architecture，指令集架构）是硬件和软件之间的**合约**。它定义了这颗芯片能执行哪些指令、每条指令做什么、操作数格式是什么。

对NVIDIA GPU来说，有两层指令集：

### PTX（Parallel Thread Execution）

```
PTX 是什么：
  NVIDIA定义的虚拟指令集
  "虚拟"意味着它不是GPU硬件直接执行的指令
  而是一个中间表示（类似Java的字节码）

你的CUDA代码编译过程：
  你写的C++/CUDA代码
       │
       ▼  (nvcc编译器前端)
  PTX代码（虚拟指令集）
       │
       ▼  (ptxas汇编器，针对具体GPU架构)
  SASS代码（真正的机器码）


一段PTX长什么样：

  // CUDA源码
  c[i] = a[i] + b[i];

  // 编译成PTX
  ld.global.f32  %f1, [%rd1];     // 从全局内存加载a[i]到寄存器f1
  ld.global.f32  %f2, [%rd2];     // 从全局内存加载b[i]到寄存器f2
  add.f32        %f3, %f1, %f2;   // f3 = f1 + f2
  st.global.f32  [%rd3], %f3;     // 把结果存回全局内存c[i]

PTX定义的内容：
  ✓ 有哪些指令（add、mul、ld、st、setp、bra...）
  ✓ 数据类型（.f16、.f32、.f64、.u32、.s64...）
  ✓ 寄存器模型（虚拟寄存器，无限多个）
  ✓ 内存空间（.global、.shared、.local、.const）
  ✓ 线程模型（threadIdx、blockIdx、gridDim）
  ✓ 特殊指令（tensor core的mma指令、NVLink的multimem指令）

为什么用虚拟指令集：
  NVIDIA每一代GPU的真实硬件指令都不一样
  V100的机器码和H100的机器码完全不兼容
  但PTX是跨代兼容的
  → 你编译成PTX，然后在运行时由驱动翻译成当前GPU的机器码
  → 这就是为什么你的CUDA程序可以在不同代GPU上运行
```

### SASS（Shader Assembly）

```
SASS 是什么：
  GPU硬件真正执行的原生机器码
  每一代GPU架构有自己的SASS格式
  
  Volta架构的SASS (sm_70)
  Ampere架构的SASS (sm_80)
  Hopper架构的SASS (sm_90)
  Blackwell架构的SASS (sm_100)
  → 互不兼容

一段SASS长什么样（H100）：

  /*0000*/  LDG.E.SYS R4, [R2] ;         // 从全局内存加载
  /*0010*/  LDG.E.SYS R5, [R6] ;         // 从全局内存加载
  /*0020*/  FADD R7, R4, R5 ;             // 浮点加法
  /*0030*/  STG.E.SYS [R8], R7 ;         // 存回全局内存

SASS的特点：
  ✓ 使用真实的物理寄存器（不是PTX的虚拟寄存器）
  ✓ 包含流水线调度信息（等待计数、屏障等）
  ✓ 反映了真实硬件的能力和限制
  
  ✗ NVIDIA从不公开SASS的完整规范
  ✗ 只能通过 cuobjdump --dump-sass 反汇编来查看
  ✗ 学术界和社区（如envytools）在持续逆向SASS格式

PTX vs SASS的关系就像：
  PTX = Java字节码     → 跨平台、公开文档化、人类可读
  SASS = x86机器码     → 特定硬件、部分不公开、底层
```

### 为什么ISA重要

```
ISA是软硬件之间的"合同"：

对软件（编译器/程序员）承诺：
  "你只要按这些指令写程序，我保证能执行"
  
对硬件（芯片设计者）约束：
  "你设计的芯片必须能正确执行这些指令"
  
这个合同一旦定了就很难改——
  改ISA意味着所有已有软件都不兼容
  所以ISA是芯片最稳定的层面
  x86指令集从1978年用到现在（47年）
  PTX从2006年用到现在（20年）
```

## 二、API / SDK（CUDA）


API（Application Programming Interface）和SDK（Software Development Kit）是NVIDIA提供给程序员的**工具箱**。如果ISA是"芯片能做什么"，那API/SDK就是"你怎么方便地使用芯片"。

```
CUDA SDK 包含的东西：

1. CUDA Runtime API（运行时接口）
   → 你最常用的那些函数
   
   cudaMalloc()          // 在GPU上分配显存
   cudaMemcpy()          // CPU和GPU之间拷贝数据
   cudaLaunchKernel()    // 启动GPU计算任务
   cudaDeviceSynchronize() // 等GPU算完
   cudaIpcGetMemHandle() // 跨进程共享GPU内存（vLLM用这个做Custom AllReduce）
   
2. CUDA Driver API（驱动级接口）
   → 更底层、更细粒度的控制
   
   cuCtxCreate()         // 创建GPU上下文
   cuModuleLoad()        // 加载GPU程序
   cuMemCreate()         // VMM虚拟内存管理（ThunderKittens用这个做NVLink通信）
   cuMulticastCreate()   // 创建NVSwitch多播对象
   
3. 编译器 nvcc
   → 把你的CUDA C++代码编译成PTX再编译成SASS
   
4. 数学库
   cuBLAS    → 矩阵运算（你的nn.Linear底层调用的就是这个）
   cuDNN     → 深度学习算子（卷积、归一化、注意力）
   cuFFT     → 快速傅里叶变换
   cuSPARSE  → 稀疏矩阵
   CUTLASS   → 高性能GEMM模板库
   
5. 通信库
   NCCL      → 多GPU集合通信
   
6. 性能分析工具
   Nsight Systems   → 系统级性能分析
   Nsight Compute   → Kernel级性能分析
   
7. 调试工具
   cuda-gdb         → GPU调试器
   compute-sanitizer → 内存检查
```

### API和ISA的关系

```
你写的代码：
  cudaMalloc(&d_ptr, size);            ← 调用CUDA API
  myKernel<<<blocks, threads>>>(d_ptr); ← 编译器把kernel编译成PTX/SASS
  cudaMemcpy(h_ptr, d_ptr, size, ...); ← 调用CUDA API

底层发生的事：
  cudaMalloc → 驱动向GPU的内存管理单元请求分配显存
  kernel启动 → 驱动向GPU提交SASS机器码 → GPU的SM开始执行指令
  cudaMemcpy → 驱动启动Copy Engine → DMA搬运数据

API是ISA的上层封装：
  ISA定义的是 "ld.global.f32 %f1, [addr]" 这种指令
  API定义的是 "cudaMemcpy(dst, src, size, kind)" 这种函数调用
  API比ISA高级得多、方便得多
  大多数程序员只和API打交道，不碰ISA
```

## 四、微架构（Microarchitecture）


微架构是ISA的**具体实现方案**——同一套ISA可以有完全不同的微架构实现。

```
ISA 说的是：  "芯片必须能执行 ADD 指令"
微架构回答的是："ADD 指令在芯片内部经过几个流水线阶段？
                用什么电路实现？延迟几个周期？吞吐量多少？"

同一个ISA，不同的微架构：
  x86 ISA → Intel Core i9（Intel的微架构）
         → AMD Ryzen 9（AMD的微架构）
         → 都能运行同样的x86程序
         → 但内部实现完全不同
         → 性能、功耗、面积都不同

对GPU来说：
  PTX ISA → V100的微架构（Volta）
         → A100的微架构（Ampere）
         → H100的微架构（Hopper）
         → 都能运行同样的PTX程序
         → 但内部实现不断演进
```

### 微架构包含什么

```
微架构 = 芯片内部"怎么做到的"的全部设计决策

┌──────────────────────────────────────────────────────────┐
│  一颗GPU的微架构包含（但不限于）：                           │
│                                                            │
│  SM内部：                                                  │
│    → 流水线设计（几级流水线、每级做什么）                     │
│    → Warp调度器的策略（轮询？贪心？动态？）                  │
│    → 寄存器文件的组织（bank数、端口数）                      │

## 五、RTL源码（Verilog / VHDL）


RTL（Register-Transfer Level，寄存器传输级）是**芯片的设计源代码**。就像软件有C++源代码一样，芯片有Verilog/VHDL源代码。这是芯片设计者每天实际编写的东西。

```
一段Verilog RTL长什么样（极度简化的例子）：

// 一个简单的32位加法器
module adder32 (
    input  wire [31:0] a,      // 输入a，32位
    input  wire [31:0] b,      // 输入b，32位
    input  wire        clk,    // 时钟信号
    output reg  [31:0] sum     // 输出sum，32位
);
    always @(posedge clk) begin  // 每个时钟上升沿
        sum <= a + b;            // sum = a + b
    end
endmodule


一个稍微复杂一点的例子——4级流水线乘法器：

module pipelined_multiplier (
    input  wire [15:0] a, b,
    input  wire        clk, rst,
    output reg  [31:0] result
);
    // 流水线寄存器
    reg [31:0] stage1, stage2, stage3;
    
    always @(posedge clk) begin
        if (rst) begin
            stage1 <= 0; stage2 <= 0; stage3 <= 0; result <= 0;
        end else begin
            stage1 <= a[7:0] * b[7:0];                    // 第1级：低位乘
            stage2 <= stage1 + (a[15:8] * b[7:0] << 8);   // 第2级：交叉乘
            stage3 <= stage2 + (a[7:0] * b[15:8] << 8);   // 第3级：交叉乘
            result <= stage3 + (a[15:8] * b[15:8] << 16);  // 第4级：高位乘
        end
    end
endmodule
```

### RTL的核心概念

```
RTL描述的是：
  → 有哪些寄存器（存储元素）
  → 每个时钟周期，数据怎么从一组寄存器流向另一组寄存器
  → 中间经过什么组合逻辑（加法器、乘法器、选择器...）

"Register-Transfer Level" 这个名字的含义：
  Register = 寄存器（存数据的）
  Transfer = 传输（数据在寄存器之间的流动）
  Level    = 这个抽象层次
  
  字面意思就是：描述"数据在寄存器之间怎么传输"的层次
```

### 真实GPU的RTL有多复杂

```
一颗H100 GPU的RTL规模（估算）：

  晶体管数量：800亿
  RTL代码行数：可能在数百万到上千万行Verilog
  设计团队：数百到上千名RTL工程师
  设计周期：3-5年
  
  对比：
  Linux内核源码：~3000万行C代码
  H100的RTL可能也在这个量级
  但每一行RTL的"密度"远高于软件代码
  因为一行RTL可能对应数千个晶体管

RTL中的一个SM可能包含：
  → warp调度器：几万行Verilog
  → 寄存器文件：几千行
  → ALU/FPU：几千行
  → Tensor Core：几万行
  → L1 Cache控制器：几万行
  → shared memory仲裁：几千行
  
NoC的RTL可能包含：
  → 每个路由器节点：几千行
  → 仲裁器：几百行
  → 虚拟通道管理：几千行
  → 流控逻辑：几百行
  → 但有几十到上百个路由器节点，加起来很大
```

### RTL的设计流程

```
芯片设计者的日常工作：

1. 写RTL代码（用Verilog或SystemVerilog）
   → 就像程序员写C++一样
   → 用的编辑器也是VS Code之类的

2. 功能仿真
   → 用仿真器（如Synopsys VCS、Cadence Xcelium）运行RTL
   → 输入测试向量，检查输出是否正确
   → 类似软件的单元测试
   → 但仿真速度极慢（一秒真实时间可能仿真几毫秒）

3. 综合（Synthesis）
   → 用工具（如Synopsys Design Compiler）
   → 把RTL"编译"成门级网表（Gate-level Netlist）
   → 把 sum <= a + b 变成具体的全加器电路连接
   → 这一步会优化面积、速度、功耗

4. 后端设计
   → 布局布线（Place and Route）
   → 这就进入物理版图的层面了
```

## 六、物理版图（GDSII）


GDSII（Graphic Data System II）是芯片的**最终物理设计文件**——它精确描述了芯片上每一层材料的每一个形状，精度到纳米级。这个文件直接送去晶圆厂（如台积电），晶圆厂根据它制造光刻掩膜版，然后制造芯片。

```
GDSII 描述的内容：

一颗芯片从截面看有十几层：

     ┌──────────────────────────┐
     │     钝化层（保护层）       │  最顶层
     ├──────────────────────────┤
     │     金属层 M15            │  最上层金属（粗线，供电/信号）
     ├──────────────────────────┤
     │     金属层 M14            │
     ├──────────────────────────┤
     │     ...（中间金属层）...    │  信号走线
     ├──────────────────────────┤
     │     金属层 M1             │  最下层金属（细线，局部连接）
     ├──────────────────────────┤
     │     接触层（Contact）      │  金属与晶体管的连接
     ├──────────────────────────┤
     │     多晶硅层（Poly）       │  晶体管的栅极
     ├──────────────────────────┤
     │     扩散区 / FinFET       │  晶体管的源极和漏极
     ├──────────────────────────┤
     │     硅衬底（Silicon）      │  最底层
     └──────────────────────────┘

GDSII文件里包含：
  每一层上的所有几何形状
  → 矩形、多边形、路径
  → 每个形状的精确坐标（纳米精度）
  → 层与层之间的对齐关系（overlay）

文件大小：
  一颗H100的GDSII可能有 数百GB 到 TB级
  包含800亿个晶体管的完整几何描述
```

### 从RTL到GDSII的过程

```
RTL源码（Verilog）
    │
    ▼  综合（Synthesis）
门级网表（Gate-level Netlist）
    │  "知道需要多少个AND门、OR门、触发器，以及它们怎么连接"
    │
    ▼  布局（Placement）
    │  "把每个门放在芯片的哪个位置"
    │  要考虑：面积最小、连线最短、热量均匀
    │
    ▼  布线（Routing）
    │  "把门与门之间的连线画出来"
    │  在十几层金属中规划路径
    │  要满足：不短路、不串扰、时序满足
    │
    ▼  签核（Sign-off）
    │  物理验证：DRC（设计规则检查）— 每条线够不够宽？间距够不够？
    │  时序验证：STA（静态时序分析）— 信号能在时钟周期内到达吗？
    │  功耗验证：能不能控制在散热范围内？
    │
    ▼
GDSII文件
    │
    ▼  送去台积电 / 三星
光刻掩膜版制造 → 晶圆制造 → 切割 → 封装 → 你买到的GPU
```

### 一个真实的物理版图片段是什么样的

```
如果你用电子显微镜看一颗芯片的某个区域：

## 七、六层之间的关系全景

```
你的PyTorch代码
   │  model = nn.Linear(4096, 4096)
   │  output = model(input)
   │
   ▼
CUDA API调用
   │  cublasSgemm(...)  ← 从cuBLAS库调用矩阵乘
   │
   ▼
PTX指令
   │  mma.sync.aligned.m16n8k16.f32.f16... ← Tensor Core矩阵乘指令
   │
   ▼
SASS机器码
   │  HMMA.16816.F32 ... ← H100实际执行的二进制指令
   │
   ▼
微架构执行
   │  warp调度器选中这条指令 → 发射到Tensor Core流水线
   │  → 4个时钟周期完成16×8×16矩阵乘
   │  → 结果写回寄存器文件
   │  → 如果需要访存，请求通过NoC发往L2/HBM
   │
   ▼
RTL层面
   │  某个触发器在时钟上升沿采样输入
   │  组合逻辑在数纳秒内完成进位传播
   │  结果稳定后被下一级触发器锁存
   │
   ▼
物理版图层面
   │  电子在FinFET的沟道中流动
   │  电信号在铜金属导线中传播
   │  经过数十亿个晶体管的开关动作
   │  最终完成一次矩阵乘法
   │
   ▼
你收到 output tensor

从你写 output = model(input) 到GPU返回结果
中间穿越了这六个层次
每一层都在做抽象——向上隐藏复杂性，向下提供精确控制
```

这六个层次就是从你写Python代码到芯片里电子流动之间的完整链路。每一层都是对下一层的抽象和封装。作为算法工程师，你工作在最上面两层（API和偶尔看看PTX）。框架开发者工作在前三层。芯片架构师工作在中间三层（白皮书、微架构、RTL）。物理设计工程师工作在最后两层（RTL、GDSII）。
