# 第16章 推理算子优化

推理引擎的宏观架构——调度器、KV Cache 管理器、分布式通信框架——决定了系统的"骨架"，但真正决定每一个 Token 生成速度的，是运行在硬件上的底层算子。一条推理请求在模型中流转时，需要经历数百次矩阵乘法、数十次注意力计算、大量的逐元素非线性变换以及归一化操作。每个算子的实现质量直接影响端到端延迟和吞吐量。在现代推理引擎中，算子优化是将硬件性能"榨干"的最后一公里。

本章从推理系统中最关键的两类计算出发——矩阵运算与注意力计算——深入分析其在 Prefill 和 Decode 两个阶段的不同性能特征与优化策略。随后，我们探讨算子融合这一通过减少访存次数来提升性能的重要技术，并分析 CUDA Graph 如何消除 CPU 端的 Kernel Launch 开销。最后，从 KTransformers 的视角出发，讨论 CPU 侧算子优化的关键技术。贯穿全章的核心思想是：推理算子优化的本质是根据算子在 Roofline 模型中的位置，选择正确的策略——计算密集型算子追求计算效率最大化，访存密集型算子追求带宽利用率最大化。

## 16.1 GEMM / GEMV 优化：推理中的矩阵运算核心

矩阵乘法是 Transformer 推理中计算量最大的算子。在一个标准的 Transformer 层中，QKV 投影、输出投影、FFN 的两个（或三个）线性变换加在一起，构成了该层绝大部分的浮点运算。理解矩阵运算在推理中的性能行为，是优化推理算子的第一步。

在推理的两个阶段中，矩阵运算呈现出截然不同的计算特征。Prefill 阶段处理整个输入序列，矩阵乘法的形状为 $(S, D) \times (D, D’)$，其中 $S$ 是序列长度，$D$ 和 $D’$ 是模型维度，这是一个典型的 GEMM（General Matrix-Matrix Multiplication）操作，三个维度都较大，属于计算密集型。Decode 阶段逐 Token 生成，矩阵乘法退化为 $(1, D) \times (D, D’)$ 的形状，本质上是 GEMV（General Matrix-Vector Multiplication），一个维度退化为 1，导致计算量大幅下降而权重的加载量不变，成为典型的访存密集型操作。这种根本性的差异要求截然不同的优化策略。

### 16.1.1 Prefill 阶段的 GEMM 优化

Prefill 阶段需要对输入序列中所有 Token 的表示同时进行矩阵变换。以一个隐藏维度 $D = 4096$、序列长度 $S = 2048$ 的场景为例，QKV 投影的矩阵乘法形状为 $(2048, 4096) \times (4096, 4096)$，其浮点运算量约为 $2 \times 2048 \times 4096 \times 4096 \approx 6.87 \times 10^{10}$ FLOPs。而权重矩阵的数据量为 $4096 \times 4096 \times 2 = 33.5$ MB（FP16），算术强度（Arithmetic Intensity）为 $\frac{6.87 \times 10^{10}}{33.5 \times 10^{6} + 2048 \times 4096 \times 2 + 2048 \times 4096 \times 2} \approx 1489$ FLOPs/Byte。在 NVIDIA H100 GPU 上（FP16 Tensor Core 算力约 989 TFLOPS，HBM 带宽约 3.35 TB/s），Roofline 模型的拐点对应的算术强度约为 $989 \times 10^{12} / (3.35 \times 10^{12}) \approx 295$ FLOPs/Byte。Prefill 阶段的 GEMM 算术强度远超这个拐点，处于计算密集区域，优化目标是最大化 Tensor Core 的利用率。

**Tiling 策略与 Tensor Core 映射。** 现代 GPU 上的高性能 GEMM 实现采用多层分块（Tiling）策略，将大矩阵乘法分解为可以映射到 GPU 层次结构的小块运算。以 CUTLASS 库为例，其典型的三级 Tiling 策略如下：在线程块（Thread Block）级别，将输出矩阵 $C$ 分割为大小约为 $128 \times 256$（或 $256 \times 128$ 等）的子块，每个线程块负责计算一个子块；在 Warp 级别，每个线程块内的 Warp 进一步将子块分割为约 $64 \times 64$ 的片段；在指令级别，每个 Warp 通过 WMMA（Warp Matrix Multiply-Accumulate）或 MMA（Matrix Multiply-Accumulate）指令，以 $16 \times 16 \times 16$（FP16）的粒度调用 Tensor Core 执行运算。

这种多级分块策略的核心目的是利用 GPU 的内存层次。外层分块从 HBM 加载数据到共享内存（Shared Memory），内层分块从共享内存加载到寄存器，最内层直接由 Tensor Core 消费。为了隐藏数据加载延迟，高性能 GEMM Kernel 通常采用软件流水线（Software Pipelining）技术：在 Tensor Core 执行当前分块的计算时，异步预取下一个分块的数据到共享内存。NVIDIA 的 Hopper 架构引入了 TMA（Tensor Memory Accelerator）硬件单元，可以由专用硬件而非 SM 上的线程来执行异步数据搬运，进一步释放了计算资源。

**数据布局与对齐。** GEMM 性能对数据在内存中的布局非常敏感。权重矩阵通常在推理初始化时被转换为对 Tensor Core 友好的布局格式。例如，CUTLASS 的 `ColumnMajorInterleaved` 布局将数据按照 Tensor Core 消费的粒度进行交错排列，使得从全局内存到共享内存的加载可以通过合并访存（Coalesced Access）实现最大带宽。此外，矩阵的维度对齐到 Tensor Core 操作粒度的整数倍（如 8 或 16）也至关重要，避免因为"尾部"处理引入额外的开销。

**Prefill 阶段的 GEMM 在推理引擎中的实现。** vLLM 和 SGLang 在 Prefill 阶段主要依赖 cuBLAS 库来执行 GEMM。cuBLAS 是 NVIDIA 官方提供的高度优化的线性代数库，其 `cublasGemmEx` 接口支持多种数据类型和算法选择。cuBLAS 内部维护了一个启发式规则（Heuristic）数据库，根据矩阵的形状和数据类型自动选择最优的 Kernel 配置。在某些场景下，推理引擎也会使用 CUTLASS 生成的定制 Kernel，特别是在需要融合额外操作（如偏置加法、激活函数）到 GEMM 中时。

对于批量 Prefill（Chunked Prefill 或连续批处理中多个 Prefill 请求同时执行的情况），一个重要的优化是将多个不同长度的序列拼接为一个大的矩阵，执行一次 GEMM 而非多次分别调用。这避免了多次 Kernel Launch 的开销，并且让 Tensor Core 有更大的矩阵块可以操作，提升了硬件利用率。

### 16.1.2 Decode 阶段的 GEMV 优化

Decode 阶段的每一步仅为一个 Token（或一个小批量的 Token）生成表示，矩阵乘法的形状变为 $(B, D) \times (D, D’)$，其中 $B$ 是批量大小。当 $B$ 较小时（例如 $B = 1$ 到 $B = 32$），这个操作的算术强度很低。以 $B = 1$、$D = D’ = 4096$（FP16）为例，计算量为 $2 \times 1 \times 4096 \times 4096 \approx 3.36 \times 10^{7}$ FLOPs，而权重数据量为 $4096 \times 4096 \times 2 = 33.5$ MB，算术强度仅为约 $1.0$ FLOPs/Byte，远低于 Roofline 拐点，处于深度访存密集区域。这意味着无论 Tensor Core 多么强大，Decode 阶段的性能天花板由 HBM 带宽决定。

**优化目标的转变。** 在访存密集区域，优化目标从"提升计算效率"转变为"提升带宽利用率"。具体而言，需要确保从 HBM 读取权重矩阵的速度尽可能接近硬件峰值带宽。这要求最小化不必要的数据读取、最大化每次内存事务的有效负载、以及通过增大批量大小来摊薄权重加载的代价。

**增大批量大小的重要性。** 连续批处理（Continuous Batching）的一个核心价值正是在此体现：通过将多个正在 Decode 的请求组成一个批次，将 GEMV 变为一个"矮胖"矩阵乘法。当批量大小从 1 增加到 $B$ 时，计算量线性增加 $B$ 倍，但权重的加载量保持不变（同一个权重矩阵服务整个批次），算术强度近似线性增加 $B$ 倍。当 $B$ 足够大时，Decode 阶段的矩阵运算可以从访存密集区域移动到计算密集区域的边界。以 H100 为例，当 $B \times D$ 的积使得算术强度超过约 295 FLOPs/Byte 时，Tensor Core 的算力开始成为瓶颈，而非带宽。对于 $D = 4096$ 的模型，这大约对应 $B \approx 295$ 的批量大小。然而，更大的批量意味着更多的 KV Cache 显存消耗和更高的单请求延迟，因此批量大小的选择需要在吞吐量和延迟之间取得平衡。

**GEMV Kernel 的实现特点。** 当批量大小确实很小时，需要专门针对 GEMV 特点优化的 Kernel。与 GEMM Kernel 不同，GEMV Kernel 的设计重点在于如何让尽可能多的线程协作读取权重矩阵的不同行（或列），同时通过归约（Reduction）操作将部分和汇聚为最终结果。典型的做法是让每个线程块负责输出向量的一段，线程块内所有线程沿着归约维度 $D$ 进行并行读取和累加，最后通过 Warp Shuffle 和共享内存归约得到结果。向量化加载指令（如 `LDG.128`，一次加载 128 位即 8 个 FP16 值）对于最大化带宽利用至关重要。

**cuBLAS 与专用 Kernel 的选择。** 在批量大小较小的 Decode 场景中，cuBLAS 的 GEMM 实现有时不是最优选择，因为 cuBLAS 的启发式规则主要针对"方形"或"大矩阵"进行调优。一些推理引擎会选择使用自定义的 GEMV Kernel 或者 cuBLAS 的 `cublasGemmStridedBatchedEx` 接口的特定配置来处理这类"极窄"矩阵乘法。

### 16.1.3 量化 GEMM Kernel（Marlin、Machete）

当模型权重被量化为低比特表示（如 INT4、INT8、FP8）后，矩阵乘法的计算模式发生了根本性变化：权重以压缩格式存储，在参与计算前需要进行反量化（Dequantization），或者直接利用硬件对低精度运算的原生支持。量化 GEMM Kernel 的设计是推理引擎中量化加速的核心。

**量化 GEMM 的基本模式。** 对于 INT4/INT8 权重量化与 FP16 激活的混合精度 GEMM，通常有两种实现策略。第一种是"反量化后计算"（Dequantize-then-Compute）：在 GEMM 的内循环中，从全局内存或共享内存加载压缩的 INT4/INT8 权重，在寄存器级别将其反量化为 FP16，然后使用 FP16 Tensor Core 执行乘累加运算。第二种是"原生低精度计算"：对于 INT8×INT8 或 FP8×FP8 的情况，可以直接使用硬件的 INT8 Tensor Core 或 FP8 Tensor Core，这在 Hopper 及更新的架构上效率极高。

**Marlin Kernel。** Marlin 是由 IST Austria 的研究者开发的一组高度优化的量化 GEMM Kernel，专门针对 GPTQ/AWQ 风格的 INT4 权重量化与 FP16 激活的矩阵乘法。Marlin 的设计目标是在量化 GEMM 中达到接近 FP16 GEMM 的硬件利用率。其核心优化策略包括以下几个方面。

首先是全局内存的异步加载流水线。Marlin 利用 NVIDIA Ampere 及更新架构的异步拷贝指令（`cp.async`），将权重数据从全局内存异步搬运到共享内存。通过构建多级软件流水线，Marlin 确保在 Tensor Core 执行当前分块计算的同时，下一分块的权重数据已经在加载过程中，从而有效隐藏内存延迟。

其次是高效的反量化策略。INT4 权重在共享内存中以压缩格式存储（每个字节存 2 个 INT4 值），在加载到寄存器时通过位操作进行解包，并与预先存储的缩放因子（Scale）和零点（Zero-Point）进行 FP16 的乘加运算完成反量化。这个过程被精心调度，使得反量化的指令与 Tensor Core 的 MMA 指令可以在 SM 的不同功能单元上并行执行。在 NVIDIA GPU 上，Tensor Core 和普通的 FP/INT 运算单元是独立的流水线，Marlin 充分利用了这一点。

第三是分块策略与并行度的精心调优。Marlin 为不同的矩阵形状和 GPU 架构（Ampere、Hopper）预设了经过详细性能调优的 Tile 大小和线程块配置。对于 Decode 阶段的小批量场景，Marlin 的 Tile 配置侧重于让更多的线程块并行工作以充分占满 GPU 的所有 SM；对于 Prefill 阶段的大矩阵，Tile 配置侧重于让每个线程块处理更大的子块以最大化 Tensor Core 利用率。

**Machete Kernel。** Machete 是 vLLM 团队与 NVIDIA 合作开发的下一代混合精度量化 GEMM Kernel 框架，旨在提供比 Marlin 更灵活、更高效的量化计算能力。Machete 基于 CUTLASS 3.x 框架构建，利用 CUTLASS 的 CuTe（CUDA Template）抽象来描述张量布局和数据搬运。与 Marlin 的手工 CUDA Kernel 不同，Machete 的设计更加模块化：量化格式的描述（比特宽度、Group Size、是否有 Zero-Point）与 GEMM 的计算逻辑分离，可以通过模板参数灵活组合。

Machete 的一个重要特性是对 Hopper 架构特性的深度利用，包括 TMA 异步加载、Warp Group MMA 指令、以及 SM-to-SM 的数据共享能力。在 vLLM 中，Machete 逐步替代 Marlin 成为某些量化配置的首选 Kernel。

**FP8 GEMM。** 对于 FP8 量化，NVIDIA Hopper 架构的 FP8 Tensor Core 提供了原生支持，理论算力是 FP16 的两倍。FP8 GEMM 可以直接使用 cuBLAS 的 FP8 路径，或者使用基于 CUTLASS 的定制 Kernel。FP8 GEMM 的挑战主要在于精度管理：E4M3 格式（4 位指数、3 位尾数）的动态范围较小，需要合适的缩放因子来避免溢出或下溢。vLLM 和 SGLang 都支持 FP8 推理，通常采用 Per-Tensor 或 Per-Channel 的静态缩放因子，在模型加载时确定。

**KLLM 的索引化矩阵乘法。** KLLM 提出了一种完全不同的范式：权重经过 K-Means 聚类后，用整数索引表示，矩阵乘法的过程变为查表和累加。在这种范式下，不再需要传统意义上的乘法运算——对于每个激活值与权重的乘积，先用权重的索引查找该激活值与各个聚类中心乘积的预计算结果，然后直接累加。这种索引化计算在通用 GPU 上难以高效实现（因为查表操作难以映射到 Tensor Core），但在 KLLM 设想的专用硬件上可以获得显著的效率提升。这种"计算范式的变革"是第 12 章详细讨论的主题，此处不再展开。

## 16.2 Attention Kernel 优化

注意力机制是 Transformer 推理中除矩阵乘法外最关键的算子。在 Prefill 阶段，注意力计算涉及长序列的 $QK^T$ 乘法、Softmax 归一化和 $\text{Score} \times V$ 乘法，计算复杂度为 $O(S^2 D)$，其中 $S$ 是序列长度。在 Decode 阶段，新生成的 Token 的 Query 需要与 KV Cache 中所有历史 Token 的 Key 和 Value 进行注意力计算。由于注意力计算的中间结果——注意力分数矩阵——的尺寸为 $O(S^2)$ 或 $O(S)$（Decode），其显存占用和访存模式对推理性能有深远影响。

### 16.2.1 FlashAttention Kernel 实现解析

FlashAttention 是推理（和训练）领域最重要的算子优化之一，其核心思想是通过分块计算（Tiling）避免将完整的 $S \times S$ 注意力分数矩阵实体化到 HBM 中，而是在 SRAM（共享内存）中完成所有中间计算。

**标准 Attention 的访存瓶颈。** 回顾标准注意力的计算流程：(1) 计算 $S = QK^T$，得到 $S \times S$ 的分数矩阵并写入 HBM；(2) 从 HBM 读取 $S$ 矩阵，计算 $P = \text{Softmax}(S)$ 并写回 HBM；(3) 从 HBM 读取 $P$ 矩阵和 $V$ 矩阵，计算 $O = PV$ 并写入 HBM。这个流程对 HBM 的读写量与 $S^2$ 成正比。当序列长度为 4096 时，FP16 下注意力分数矩阵就占据 $4096 \times 4096 \times 2 = 32$ MB（每个注意力头），远超 GPU 上 SM 的共享内存容量（通常为 128 KB 到 228 KB）。大量的 HBM 读写使得标准注意力计算成为访存密集型操作，即使计算量并不大。

**FlashAttention-1 的核心算法。** FlashAttention-1 通过 Online Softmax 技巧实现了分块注意力计算。其核心洞察是：Softmax 可以增量地计算——在处理 $K$ 矩阵的新分块时，可以根据新分块的最大值更新之前分块的 Softmax 结果，而无需回头重新读取之前的分数。

具体而言，FlashAttention-1 的 Kernel 按照以下方式工作。外层循环遍历 $K$ 和 $V$ 的分块（沿序列维度分割），内层循环遍历 $Q$ 的分块。对于每一对 $Q$ 分块和 $K$ 分块：(a) 从 HBM 加载 $Q$ 分块和 $K$ 分块到共享内存；(b) 在共享内存/寄存器中计算局部注意力分数 $S_{ij} = Q_i K_j^T$；© 更新该 $Q$ 分块的运行最大值 $m$ 和运行求和 $\ell$（这是 Online Softmax 的核心）；(d) 根据更新后的最大值重新缩放之前的累积输出，并加上当前分块的贡献；(e) 处理完所有 $K$ 分块后，对最终输出做一次归一化，得到正确的 Softmax 注意力结果。

这种方法的HBM 读写量从 $O(S^2)$ 降低到 $O(S)$（只需要读取 $Q$、$K$、$V$ 并写出 $O$，每个都是 $O(S \times D)$），带来了显著的速度提升——并非因为减少了计算量，而是因为大幅减少了对 HBM 的访问。

**FlashAttention-2 的优化。** FlashAttention-2 在算法层面保持不变，但在工程实现上进行了多处重要优化。第一，将外层循环改为遍历 $Q$ 的分块而非 $K$ 的分块，这样每个线程块处理输出 $O$ 的不同行，避免了线程块之间对输出的竞争写入，消除了同步开销。第二，减少了非矩阵乘法运算（如 Softmax 的指数、求和）在整体计算中的比例，更好地利用了 Tensor Core。FlashAttention-2 通过将工作分配给更少的 Warp（通常 4 个或 8 个），让每个 Warp 做更多的工作，提升了 Warp 级别的并行效率。第三，FlashAttention-2 在长序列上还支持沿序列维度的并行——当 $Q$ 的分块数量不足以填满所有 SM 时，可以沿 $K$/$V$ 方向进一步并行化。

**FlashAttention-3 的进一步演进。** FlashAttention-3 专门针对 NVIDIA Hopper 架构优化，利用了三个关键硬件特性：(a) WGMMA（Warp Group Matrix Multiply-Accumulate）指令，支持直接从共享内存到 Tensor Core 的数据流，减少了寄存器压力；(b) TMA 异步数据搬运，将全局内存到共享内存的数据加载完全卸载到专用硬件单元；© FP8 Tensor Core 支持，可以在注意力计算中使用 FP8 精度来换取更高的吞吐量。FlashAttention-3 还实现了更精细的异步流水线，将数据加载、Softmax 计算和矩阵乘法三个阶段重叠执行，进一步压缩了端到端延迟。

**Decode 阶段的 FlashAttention 行为。** 在 Decode 阶段，$Q$ 的长度为 1（或批量大小 $B$），而 $K$/$V$ 的长度为整个 KV Cache 的长度。此时注意力计算实际上退化为一个向量与矩阵的点积加 Softmax 的操作，其访存模式完全不同于 Prefill。FlashAttention 的原始实现主要针对 Prefill 阶段优化，对于 Decode 阶段，推理引擎通常使用专门设计的 Decode Attention Kernel（如 FlashDecoding、FlashInfer 的 Decode Kernel），这些 Kernel 针对"Query 很短、KV 很长"的形状进行了专门优化。

### 16.2.2 FlashInfer：推理专用 Attention Kernel 库

FlashInfer 是一个专为 LLM 推理场景设计的注意力 Kernel 库，由 CMU 的研究者开发，是 SGLang 推理引擎默认使用的注意力后端，在 vLLM V1 中也被集成为可选后端。FlashInfer 的设计哲学是为推理中各种复杂的注意力模式提供统一且高效的 Kernel 实现。

**推理场景的特殊需求。** 与训练不同，推理中的注意力计算面临以下特殊挑战：(a) KV Cache 是分页存储的（PagedAttention），物理上不连续；(b) 同一批次中不同请求的 KV Cache 长度差异很大；© Decode 阶段的 Query 长度极短（通常为 1），需要最大化 KV Cache 的并行读取效率；(d) 需要支持 GQA、MLA 等不同的注意力变体。FlashInfer 针对这些需求提供了专门的解决方案。

**Decode Attention 的 FlashDecoding 优化。** Decode 阶段注意力的核心瓶颈是 KV Cache 的读取。对于一个 KV Cache 长度为 $L$ 的请求，需要读取 $L$ 个 Key 向量和 $L$ 个 Value 向量，但 Query 只有一个。如果采用 FlashAttention 的原始策略，只有一个（或很少几个）线程块负责这个请求，无法充分利用 GPU 的并行度。

FlashDecoding（最初由 Tri Dao 提出，FlashInfer 中有改进实现）的解决方案是将长 KV Cache 沿序列维度分割为多个分块，每个分块由一个独立的线程块并行处理，各自计算局部的注意力分数、局部最大值和局部加权求和。所有分块处理完毕后，通过一个轻量的归约 Kernel 将各分块的结果合并（利用 Online Softmax 的增量更新公式）。这样，一个请求的 Decode Attention 可以利用多个 SM 并行处理，大幅降低了延迟。

**Ragged Tensor 与 Paged KV Cache 的统一抽象。** FlashInfer 引入了 Ragged Tensor 的概念来处理批次中不同请求长度不同的情况。对于 Prefill 阶段，多个请求的 $Q$、$K$、$V$ 被拼接为一个连续的张量，并通过一个索引数组记录每个请求的起始位置和长度。对于 Decode 阶段的 Paged KV Cache，FlashInfer 支持通过 Page Table（Block Table）来间接寻址物理上不连续的 KV Cache 块。Kernel 在执行注意力计算时，根据 Page Table 动态地从不同的物理块中读取 Key 和 Value，对上层应用透明。

FlashInfer 对 Paged KV Cache 的注意力 Kernel 做了专门优化，使得页表查找的开销被异步数据加载所掩盖。具体地，在加载一个 KV Cache 块的数据时，可以同步进行下一个块的页表查找和地址计算，确保流水线不因间接寻址而中断。

**Plan-Execute 两阶段 API。** FlashInfer 采用 Plan-Execute 的两阶段 API 设计。在 Plan 阶段（通常在 CPU 上执行），根据当前批次的元信息（每个请求的 Query 长度、KV Cache 长度、Page Table）预计算 GPU Kernel 需要的辅助数据结构，如每个线程块处理的请求分配、并行分割策略等。在 Execute 阶段（在 GPU 上执行），直接使用 Plan 阶段的输出执行注意力计算，避免在 GPU Kernel 内部做复杂的控制流决策。这种设计使得 GPU Kernel 的代码路径更简单、性能更可预测。

**JIT 编译与特化。** FlashInfer 支持通过 JIT（Just-In-Time）编译生成针对特定配置特化的 Kernel。例如，对于特定的注意力头数、头维度、页大小组合，FlashInfer 可以在首次使用时编译出一个特化的 Kernel，其中所有这些参数都是编译时常量，使得编译器可以进行更激进的优化（如循环展开、常量折叠、寄存器分配优化）。虽然 JIT 编译有首次编译的延迟，但编译结果会被缓存，后续调用直接使用编译好的 Kernel。

### 16.2.3 Paged KV Cache 的 Attention Kernel 适配

Paged KV Cache（第 5 章详述）通过将 KV Cache 分为固定大小的物理块（Block）来消除内存碎片化，但这也给注意力 Kernel 的实现带来了额外的复杂性。

**间接寻址的挑战。** 在没有分页的情况下，一个请求的 KV Cache 存储在一段连续的内存区域中，Attention Kernel 可以通过简单的指针偏移量来访问任意位置的 Key 和 Value。引入分页后，KV Cache 的逻辑位置（Token 在序列中的位置）和物理位置（在 GPU 显存中的地址）之间需要通过 Block Table 进行映射。对于逻辑位置 $p$ 的 Token，其所在的物理块编号为 $\text{BlockTable}[\lfloor p / B_s \rfloor]$（$B_s$ 是每个块的容量），块内偏移为 $p \mod B_s$。

这种间接寻址增加了内存访问的延迟，因为在读取实际的 KV 数据之前，需要先从 Block Table 中查询物理块编号。如果 Block Table 本身不在缓存中，这会引入额外的内存读取。

**Kernel 实现中的适配策略。** vLLM 最初的 PagedAttention Kernel 是在 FlashAttention 的框架上修改而来，在内层循环中增加了 Block Table 查询的逻辑。其基本流程是：对于 KV Cache 的每一个分块，先从 Block Table 中查找该分块对应的物理块地址，然后从该物理地址加载 Key 和 Value 数据到共享内存，后续的注意力计算逻辑与标准 FlashAttention 一致。

FlashInfer 对此做了更深入的优化。在 Plan 阶段，FlashInfer 会预先将分散的 Page Table 信息整理为对 GPU Kernel 友好的格式——例如，将多个请求的 Page Table 拼接为一个连续数组，并预计算每个线程块需要访问的物理块范围。在 Execute 阶段的 Kernel 中，利用 TMA 或 `cp.async` 指令的异步加载能力，在处理当前物理块的注意力计算时，预取下一个物理块的数据。这样，Page Table 查询和数据预取的延迟被隐藏在计算之后。

**块大小的选择。** KV Cache 物理块的大小（Block Size）直接影响注意力 Kernel 的性能。较大的块（如 64 或 128 个 Token）减少了 Page Table 的条目数量和间接寻址的次数，并且有利于合并内存访问；但可能导致更大的内部碎片（一个请求最后一个块的未使用空间）。较小的块（如 16 个 Token）减少了碎片但增加了间接寻址的开销。在实践中，vLLM 默认使用 16 作为块大小，SGLang / FlashInfer 通常使用 1 的"无分页"模式配合 Token-级别的内存管理，或使用较小的块大小。选择取决于具体的注意力 Kernel 实现和硬件特性。

**vLLM V1 的转变。** vLLM V1 架构中，注意力 Kernel 的实现经历了重要变化。V0 架构使用自研的 PagedAttention Kernel，V1 架构则逐步迁移到 FlashInfer 作为默认的注意力后端。这一转变的原因在于 FlashInfer 的 Kernel 在多种场景下表现出更好的性能，特别是在 Decode 阶段和长序列场景中。同时，FlashInfer 作为一个独立维护的高质量 Kernel 库，减轻了 vLLM 团队在底层 Kernel 优化上的维护负担，使其可以更专注于系统层面的优化。

## 16.3 算子融合（Operator Fusion）

算子融合是将多个独立的算子合并为一个复合 Kernel 执行的优化技术。其核心收益是减少中间结果的 HBM 读写次数。在一个典型的 Transformer 层中，如果每个算子都独立执行，每个算子都需要将输出写入 HBM，下一个算子再从 HBM 读取输入。对于访存密集型的逐元素算子（如激活函数、归一化），这种反复的 HBM 读写往往比计算本身更耗时。通过融合，多个算子在一次 Kernel 调用中完成，中间结果保留在寄存器或共享内存中，大幅减少了 HBM 访问。

### 16.3.1 常见融合模式：QKV Fusion、Gate-Up Fusion、SwiGLU Fusion

**QKV 投影融合。** 在标准的 Multi-Head Attention 中，Query、Key、Value 三个投影是三个独立的线性变换：$Q = XW_Q$，$K = XW_K$，$V = XW_V$。这三个操作共享相同的输入 $X$，但使用不同的权重矩阵。朴素实现需要三次 GEMM 调用和三次 $X$ 的 HBM 读取。QKV 融合将三个权重矩阵沿输出维度拼接为一个大矩阵 $W_{QKV} = [W_Q; W_K; W_V]$，然后通过一次 GEMM 计算 $[Q; K; V] = XW_{QKV}$。这不仅减少了 Kernel Launch 次数（从 3 次到 1 次），还减少了输入 $X$ 的 HBM 读取次数（从 3 次到 1 次），并且一个更大的 GEMM 通常比三个较小的 GEMM 能更好地利用 Tensor Core。

对于 GQA（Grouped-Query Attention），$W_K$ 和 $W_V$ 的维度小于 $W_Q$（因为 KV 头数少于 Query 头数）。融合时需要处理这种不对称性，通常通过在拼接矩阵中添加适当的填充（Padding）或使用分组 GEMM（Grouped GEMM）来实现。

**Gate-Up Fusion 与 SwiGLU Fusion。** 现代 Transformer 模型（如 LLaMA、DeepSeek）的 FFN 层通常采用 SwiGLU 激活函数，其计算公式为：

$$\text{FFN}(x) = (\text{SiLU}(xW_{\text{gate}}) \odot xW_{\text{up}}) W_{\text{down}}$$

其中 $\text{SiLU}(z) = z \cdot \sigma(z)$ 是 Sigmoid Linear Unit，$\odot$ 表示逐元素乘法。这个计算包含两个并行的线性变换（Gate 和 Up），一个非线性激活（SiLU），一个逐元素乘法，以及一个输出线性变换（Down）。

Gate-Up Fusion 将 $W_{\text{gate}}$ 和 $W_{\text{up}}$ 拼接为一个矩阵 $W_{\text{gate-up}} = [W_{\text{gate}}; W_{\text{up}}]$，通过一次 GEMM 同时计算 Gate 和 Up 投影的结果。这与 QKV 融合的思路完全一致。

进一步的 SwiGLU Fusion 将 SiLU 激活和逐元素乘法融合到 Gate-Up GEMM 的 Epilogue（收尾阶段）中。在 CUTLASS 框架中，GEMM 的 Epilogue 是矩阵乘法完成后对输出元素进行后处理的阶段，此时数据还在寄存器中。通过自定义 Epilogue，可以在结果写入 HBM 之前直接执行 SiLU 和逐元素乘法，避免了一次完整的中间结果 HBM 读写。然而，这种深度融合的实现复杂度较高，需要修改 GEMM Kernel 的代码生成逻辑。在当前的 vLLM 和 SGLang 实现中，Gate-Up GEMM 的融合是标准做法，但 SiLU 与逐元素乘法的 Epilogue 融合通常通过 Torch.compile 或 Triton 自定义 Kernel 实现，而不是直接修改 CUTLASS Kernel。

### 16.3.2 LayerNorm + Residual 融合

在 Transformer 的每一层中，LayerNorm（或 RMSNorm）和残差连接是必经之路。以 Pre-Norm 架构为例，一个典型的计算模式是：

$$h = \text{Attention}(\text{RMSNorm}(x)) + x$$

其中 RMSNorm 的计算公式为：

$$\text{RMSNorm}(x) = \frac{x}{\sqrt{\frac{1}{D}\sum_{i=1}^{D} x_i^2 + \epsilon}} \cdot \gamma$$

RMSNorm 和残差加法都是逐元素的操作，各自的计算量很小但需要读写整个隐藏状态向量。如果分别执行，RMSNorm 需要读取 $x$、写出 $\text{RMSNorm}(x)$，残差加法需要读取 Attention 输出和 $x$、写出 $h$。

**融合策略。** RMSNorm + Residual Add 融合 Kernel 的工作流程如下：一次性从 HBM 读取输入 $x$ 和 Attention 子层的输出（或上一步的结果），在寄存器中同时完成 (a) 计算 $x$ 的 RMS 统计量、(b) 对 $x$ 进行归一化（为下一个子层的输入做准备）、© 将 Attention 子层的输出与 $x$ 相加得到残差结果，然后将归一化的输入和残差结果分别写回 HBM。这样，$x$ 只从 HBM 读取一次，而非朴素实现中的多次。

在实践中，这种融合 Kernel 通常以 Triton 或 CUDA 编写。vLLM 和 SGLang 都使用了类似的 Fused RMSNorm Kernel。在 SGLang 的 Torch.compile 集成中，PyTorch 编译器的 TorchInductor 后端可以自动识别并融合这类模式，生成高效的 Triton Kernel。

### 16.3.3 RoPE 嵌入的融合处理

旋转位置编码（Rotary Position Embedding，RoPE）是现代 Transformer 模型（LLaMA、DeepSeek 等）的标准位置编码方式。RoPE 对 Query 和 Key 向量的每一对相邻元素应用一个与位置相关的旋转变换：

$$\begin{pmatrix} q’_{2i} \ q’_{2i+1} \end{pmatrix} = \begin{pmatrix} \cos(m\theta_i) & -\sin(m\theta_i) \ \sin(m\theta_i) & \cos(m\theta_i) \end{pmatrix} \begin{pmatrix} q_{2i} \ q_{2i+1} \end{pmatrix}$$

其中 $m$ 是 Token 的位置，$\theta_i = 10000^{-2i/D}$ 是频率参数。这个操作是逐元素的（每对元素独立处理），计算量很小。

**融合到 QKV 投影之后。** 在朴素实现中，QKV 投影的输出写入 HBM，然后 RoPE Kernel 从 HBM 读取 $Q$ 和 $K$，应用旋转变换，再写回 HBM。融合策略是将 RoPE 嵌入到 QKV 投影 GEMM 的 Epilogue 中，或者更常见地，在 Attention Kernel 的开头执行 RoPE——即在 Attention Kernel 内部，读取 $Q$ 和 $K$ 后立即在寄存器中应用 RoPE 变换，然后进行后续的注意力计算。FlashInfer 的注意力 Kernel 支持内置的 RoPE 应用，SGLang 和 vLLM 在某些代码路径中也采用了这种方式。

**频率参数的预计算。** RoPE 所需的 $\cos(m\theta_i)$ 和 $\sin(m\theta_i)$ 值通常在推理服务启动时预计算好，存储为一个查找表。在 Kernel 执行时，根据 Token 位置 $m$ 直接从表中读取，避免了昂贵的三角函数计算。

## 16.4 CUDA Graph

CUDA Graph 是 NVIDIA 提供的一种将多个 CUDA 操作（Kernel Launch、内存拷贝等）捕获为一个静态的执行图，然后一次性提交给 GPU 执行的机制。在 LLM 推理中，CUDA Graph 的主要价值是消除 CPU 端的 Kernel Launch 开销。

### 16.4.1 消除 Kernel Launch 开销

**Kernel Launch 开销的来源。** 在 GPU 编程中，每次调用一个 CUDA Kernel 时，CPU 端的 CUDA 驱动需要完成一系列工作：验证参数、构造 Kernel 描述符、将描述符提交到 GPU 的命令队列。单次 Kernel Launch 的 CPU 端开销通常在 3 到 10 微秒之间。虽然看起来很小，但 Transformer 推理的一步（一个 Token 的 Decode）涉及几十到上百个 Kernel 调用（线性变换、注意力、归一化、激活函数、采样等），这些 Launch 开销累积起来可能达到数百微秒甚至毫秒级别。

**Decode 阶段的影响尤为突出。** 在 Decode 阶段，由于批量大小通常不大，每个 Kernel 的 GPU 执行时间可能只有几十到几百微秒。此时，CPU 端的 Kernel Launch 开销与 GPU 端的实际计算时间处于同一量级，Kernel Launch 开销在总延迟中的占比可能高达 20%—50%。相比之下，在 Prefill 阶段，由于序列较长，单个 GEMM Kernel 的执行时间可能达到毫秒级，Kernel Launch 开销的占比较小。

**CUDA Graph 的工作原理。** CUDA Graph 通过两个阶段工作：捕获（Capture）和回放（Replay）。在捕获阶段，应用程序按正常方式执行一系列 CUDA 操作，但这些操作不会真正执行，而是被记录到一个图结构中。图的节点代表 CUDA 操作（Kernel Launch、内存拷贝），边代表操作之间的依赖关系。捕获完成后，调用 `cudaGraphInstantiate` 将图编译为一个可执行的图实例。在回放阶段，调用 `cudaGraphLaunch` 一次性将整个图提交给 GPU，GPU 内部按照图的拓扑顺序依次执行所有操作，无需 CPU 的逐一干预。这样，无论图中包含多少个 Kernel，CPU 端只产生一次 Launch 开销。

### 16.4.2 vLLM / SGLang 中的 CUDA Graph 使用

vLLM 和 SGLang 都在 Decode 阶段大量使用 CUDA Graph 来加速推理。然而，将 CUDA Graph 应用到推理引擎并非简单地将整个推理流程捕获为一个图，原因在于 Decode 阶段的计算存在动态性——每一步的批量大小可能不同（由于连续批处理中请求的动态加入和退出），KV Cache 的长度在增长，Page Table 在变化。

**预捕获多个图。** vLLM 和 SGLang 的策略是在推理引擎初始化时（或首次遇到特定配置时），针对不同的批量大小预先捕获多个 CUDA Graph。例如，预先捕获批量大小为 1、2、4、8、16、32、64、128、256 等的图。在运行时，根据当前的实际批量大小选择最接近的（不小于实际大小的）预捕获图进行回放。对于实际批量大小与图的批量大小不一致的情况，通过填充（Padding）来处理——将实际的输入张量填充到图期望的大小，图执行完毕后只取有效部分的输出。

**输入/输出张量的更新。** CUDA Graph 在捕获时记录了所有 Kernel 的参数（包括输入/输出张量的内存地址和值），在回放时需要更新这些参数以反映当前步的实际数据。vLLM 和 SGLang 通过以下方式处理：预先分配固定大小的输入/输出缓冲区作为 CUDA Graph 的输入/输出；在每一步开始前，将当前的输入数据拷贝到这些固定缓冲区中；图回放后，从固定缓冲区中读取输出。对于 KV Cache 地址、Page Table 等需要每步更新的指针信息，使用 CUDA Graph 的 `cudaGraphKernelNodeSetParams` 接口或通过 CUDA Memory Pool 和间接指针来更新。

**vLLM V1 的 CUDA Graph 实现。** vLLM V1 架构对 CUDA Graph 的集成进行了简化。在 V1 中，CUDA Graph 默认启用（除非使用 `--enforce-eager` 参数强制使用 Eager 模式），ModelRunner 在初始化阶段根据配置的最大批量大小范围预热并捕获多个图。V1 还优化了图选择的逻辑，减少了不必要的填充开销。

**SGLang 的 CUDA Graph 使用。** SGLang 同样默认启用 CUDA Graph，并且在 Torch.compile 集成的背景下，CUDA Graph 的行为与 PyTorch 编译器的图捕获能力相互配合。SGLang 的实现中，CUDA Graph 的捕获范围覆盖了从 Model Forward 到 Sampling 的完整路径，确保最大程度地消除 Launch 开销。

### 16.4.3 动态形状与 CUDA Graph 的适配

CUDA Graph 的最大限制是其静态性：图在捕获时的所有张量形状、Kernel 参数都被固化。然而，推理过程天然是动态的——批量大小、序列长度、KV Cache 长度都在变化。解决这种静态性与动态性之间矛盾的策略有以下几种。

**桶化（Bucketing）。** 前面提到的"预捕获多个不同批量大小的图"就是桶化策略。将可能的批量大小映射到有限的几个"桶"中，每个桶对应一个预捕获的图。桶的选择需要平衡两方面：桶越多，填充浪费越少，但显存占用越大（每个图都会占用一定的显存用于内部缓冲区）；桶越少，显存节省，但填充浪费增加。vLLM 和 SGLang 通常使用几何级数（1, 2, 4, 8, …）的桶间距，在小批量时提供精细的粒度，在大批量时容忍适度的填充。

**CUDA Graph 与动态张量形状的处理。** 对于 KV Cache 长度的变化，由于注意力 Kernel（如 FlashInfer）接受 KV Cache 的元数据（如 Page Table、序列长度数组）作为输入，这些元数据可以在 CUDA Graph 回放前通过内存拷贝更新。关键在于 Kernel 本身的逻辑需要根据这些运行时参数动态调整行为（如循环次数），而这在 CUDA Graph 中是允许的——图中 Kernel 的代码逻辑不变，只是输入数据不同。

**CUDA Graphs 与 Torch.compile 的协同。** 当 SGLang 使用 Torch.compile 时，PyTorch 编译器可以生成更有利于 CUDA Graph 捕获的代码。Torch.compile 的 TorchInductor 后端会将多个小 Kernel 融合为少数几个大 Kernel，减少了图中的节点数量。同时，PyTorch 2.x 引入了 `torch.compile` 与 CUDA Graph 的原生协同模式——`torch.compile(mode="reduce-overhead")` 会自动将编译后的函数封装为 CUDA Graph。

**不适合 CUDA Graph 的场景。** 以下情况通常不使用 CUDA Graph：(a) Prefill 阶段的序列长度变化大，GEMM 的矩阵形状差异显著，预捕获所有可能的形状不现实；(b) 涉及复杂控制流的场景（如投机解码中的接受/拒绝分支）；© 调试阶段——CUDA Graph 使得 GPU 操作不透明，难以使用 profiler 定位单个 Kernel 的性能问题。vLLM 的 `--enforce-eager` 参数和 SGLang 的类似选项允许用户禁用 CUDA Graph 以便于调试。

## 16.5 CPU 算子优化（KTransformers 视角）

KTransformers 的核心理念是将 MoE 模型中的专家计算卸载到 CPU 上执行，从而在有限的 GPU 资源下实现超大模型的推理。这一架构的成败很大程度上取决于 CPU 侧算子的执行效率。传统观点认为 CPU 的计算能力远不如 GPU，但现代服务器级 CPU（如 Intel Xeon Scalable 第四代及以后的型号）配备了专用的矩阵计算加速指令集和大容量高带宽内存，使得在 CPU 上高效执行量化矩阵运算成为可能。

### 16.5.1 AMX 指令集的矩阵计算优化

**AMX 架构概述。** Intel Advanced Matrix Extensions（AMX）是 Intel 在第四代 Xeon Scalable 处理器（代号 Sapphire Rapids）中引入的矩阵计算加速指令集。AMX 引入了一组新的硬件寄存器——Tile 寄存器（`tmm0` 到 `tmm7`，共 8 个），每个 Tile 可以容纳一个最大 $16 \times 64$ 字节的二维数据块。AMX 还引入了对应的指令：`TILELOADD`（从内存加载数据到 Tile）、`TILESTORED`（从 Tile 存储数据到内存）、`TDPBF16PS`（BF16 矩阵乘加）、`TDPBSSD`（INT8 矩阵乘加）等。

以 `TDPBSSD` 指令为例，它可以执行一个 $16 \times 64$ 的 INT8 矩阵与一个 $64 \times 16$ 的 INT8 矩阵的乘法，并将结果累加到一个 $16 \times 16$ 的 INT32 矩阵上，单条指令完成 $16 \times 64 \times 16 = 16384$ 次 INT8 乘加运算。对于一个运行在 2 GHz 的 CPU 核心，每周期可以发射一条 AMX 指令，理论 INT8 吞吐量约为 $16384 \times 2 \times 10^9 = 32.8$ TOPS（每核心）。一个 56 核的 Xeon 处理器的总 INT8 吞吐量可达约 1800 TOPS，这虽然仍远低于 NVIDIA H100 GPU 的 INT8 算力（约 3958 TOPS），但对于 MoE 模型中单个专家的计算量而言已经是足够的。

**KTransformers 中的 AMX 矩阵计算。** KTransformers 将 MoE 模型的专家权重存储在 CPU 内存中（通常以 INT4/INT8 量化格式），当 Gate 网络选择了需要执行的专家后，KTransformers 在 CPU 侧执行专家的矩阵运算。其关键优化包括以下方面。

第一是量化格式与 AMX 的适配。专家权重以 INT4 或 INT8 量化格式存储。对于 INT4 格式，需要先将 INT4 解包为 INT8（每个 INT4 值占据一个 INT8 的低 4 位），然后使用 AMX 的 INT8 指令执行计算。解包操作可以通过 SIMD 指令高效完成。对于 GGUF 中常见的 Q4_K_M 等量化格式，KTransformers 开发了专门的解包例程将其转换为 AMX 友好的布局。

第二是 Tile 布局优化。AMX 的 Tile 寄存器要求数据按照特定的布局排列。KTransformers 在模型加载时预先将权重矩阵重排为 AMX 友好的布局（即与 Tile 寄存器的 $16 \times 64$ 字节对齐的分块格式），避免在运行时进行布局转换。

第三是多核并行。一个专家的矩阵运算通过 OpenMP 或自定义线程池分配到多个 CPU 核心上并行执行。每个核心负责输出矩阵的不同行，利用各自的 AMX 单元独立计算。核心间不需要通信（因为不存在对输出的竞争写入），并行效率很高。

### 16.5.2 SIMD 向量化与内存预取

除 AMX 指令外，CPU 上还有其他重要的计算加速手段，用于处理非矩阵乘法的操作（如激活函数、归一化、逐元素运算）以及数据准备工作。

**AVX-512 向量化。** AVX-512 是 Intel 的 512 位 SIMD（Single Instruction, Multiple Data）指令集，一条 AVX-512 指令可以同时处理 16 个 FP32 值、32 个 FP16/BF16 值或 64 个 INT8 值。在 KTransformers 中，除了专家的矩阵乘法使用 AMX 外，其他逐元素操作（如 SiLU 激活函数、RMSNorm、残差加法、反量化中的缩放和偏置操作）都使用 AVX-512 向量化实现。

以 SiLU 激活函数为例，$\text{SiLU}(x) = x \cdot \sigma(x) = x / (1 + e^{-x})$，一个 AVX-512 向量化实现可以同时计算 16 个 FP32 值的 SiLU。指数函数 $e^{-x}$ 的向量化实现通常基于多项式近似（如 Remez 算法生成的逼近多项式），精度可以根据需要调整。

**内存预取（Prefetching）。** CPU 的内存层次结构包括 L1/L2/L3 Cache 和主内存（DDR）。L1 Cache 的访问延迟约 1 纳秒，DDR 的访问延迟约 80-100 纳秒，差距约两个数量级。内存预取技术通过提前将即将使用的数据从主内存加载到 Cache 中，来隐藏这种延迟差异。

在 KTransformers 的矩阵计算中，内存预取至关重要。当 CPU 核心正在使用 AMX 指令处理当前分块的权重数据时，同时使用 `_mm_prefetch` 指令将下一个分块的权重数据预取到 L1 或 L2 Cache 中。预取的距离（提前多少个分块）需要根据主内存的延迟和计算的耗时仔细调整——预取太早会导致数据在使用前被其他数据从 Cache 中驱逐，预取太晚则无法完全隐藏内存延迟。

KTransformers 的实现中，对于 DeepSeek-V3 模型的专家权重访问，由于每个专家的参数量较大（约数百 MB），不可能将完整专家参数保持在 Cache 中。因此，预取策略采用了流水线式的分块加载：将专家的权重矩阵分为多个小块，每个小块的大小选择使其能够放入 L2 Cache。在处理第 $i$ 块时预取第 $i+1$ 块，形成一个稳定的流水线。

### 16.5.3 NUMA 感知的内存访问优化

现代多路服务器（双路或多路 CPU）采用 NUMA（Non-Uniform Memory Access）架构：每个 CPU 插槽（Socket）有自己的本地内存通道，访问本地内存的延迟低于访问远程 Socket 的内存。在双路系统中，跨 Socket 的内存访问延迟可能比本地访问高 50%—100%。

**NUMA 对推理性能的影响。** KTransformers 通常部署在多路服务器上，以利用足够的 CPU 核心数和内存容量（DeepSeek-V3 671B 模型即使量化后仍需要约 300-400 GB 的 CPU 内存来存储专家参数）。如果专家参数的内存分配和线程调度不是 NUMA 感知的，可能出现以下问题：线程在 Socket 0 上运行，但需要访问分配在 Socket 1 内存中的权重数据，导致跨 Socket 的内存访问成为瓶颈。

**NUMA 感知的优化策略。** KTransformers 采用以下策略来优化 NUMA 访问。首先是内存绑定（Memory Binding）：使用 `numactl` 或 `libnuma` 库将专家参数的内存分配绑定到特定 Socket 的本地内存上。例如，将编号为偶数的专家参数分配到 Socket 0 的内存，奇数专家分配到 Socket 1 的内存。

其次是线程绑定（Thread Affinity）：将计算线程绑定到与其需要访问的数据所在的同一 Socket 上的核心。当 Gate 网络选择了一个需要执行的专家时，将该专家的计算任务分配给与其参数所在 Socket 对应的线程池。

第三是数据局部性最大化：在设计计算的分块策略时，确保每个线程在一段时间内只访问其本地内存中的数据，减少跨 Socket 的内存流量。

第四是内存交错（Memory Interleaving）：在某些场景下，如果专家的选择模式难以预测且频繁变化，可以采用内存交错策略——将数据按页面（通常 4 KB）轮流分配到不同 Socket 的内存上。虽然每次访问都可能有约一半的概率是远程访问，但平均性能是可预测且稳定的，避免了最坏情况下所有访问都是远程访问的情况。

**CPU-GPU 数据传输的 NUMA 考虑。** 在 KTransformers 的 CPU-GPU 异构架构中，GPU 通过 PCIe 连接到某一个 CPU Socket。从 GPU 的角度看，它与某个 Socket 的"距离"更近。当需要在 CPU 和 GPU 之间传输数据时（如将 CPU 上计算的专家输出传回 GPU），应确保数据位于与 GPU 直连的 Socket 的本地内存上，以获得最大的 PCIe 带宽。如果数据位于远程 Socket 的内存中，PCIe 传输需要先通过 CPU 间的互联（如 UPI/CXL）将数据从远程 Socket 搬运到本地 Socket，再通过 PCIe 发送给 GPU，带宽和延迟都会受到影响。

## 本章小结

本章深入探讨了大模型推理中底层算子优化的关键技术。推理算子优化的核心思想可以归结为一个简洁的原则：理解每个算子在 Roofline 模型中的位置，然后针对性地优化瓶颈资源。

对于 GEMM/GEMV 运算，Prefill 阶段的大矩阵乘法处于计算密集区域，需要通过精细的 Tiling、软件流水线和 Tensor Core 映射来最大化算力利用率；Decode 阶段的小批量矩阵运算处于访存密集区域，需要最大化 HBM 带宽利用并通过增大批量大小来改善算术强度。量化 GEMM Kernel（Marlin、Machete）通过在减少权重数据量的同时高效利用 Tensor Core，在两个阶段都带来了显著的加速。

对于注意力计算，FlashAttention 系列通过分块计算和 Online Softmax 消除了 $O(S^2)$ 的 HBM 访问瓶颈，FlashInfer 则针对推理场景的特殊需求（Paged KV Cache、变长批次、Decode 并行化）提供了专门优化的 Kernel。算子融合通过合并多个小算子为一个 Kernel，减少了中间结果的 HBM 读写。CUDA Graph 通过消除 CPU 端的 Kernel Launch 开销，在 Decode 阶段带来了可观的延迟降低。

从 KTransformers 的视角，CPU 侧算子优化展示了异构计算的另一种可能性：通过 AMX 矩阵计算指令、AVX-512 向量化、精心的内存预取和 NUMA 感知调度，现代 CPU 可以高效地承担 MoE 模型中专家计算的任务，为在资源受限条件下部署超大模型开辟了道路。

这些底层优化技术看似独立，但在实际的推理引擎中它们相互交织、协同发挥作用。理解这些技术，不仅有助于在使用现有推理引擎时做出正确的配置选择，更为开发新的优化方案和适配新硬件平台奠定了必要的知识基础。在下一章中，我们将从单个算子上升到计算图层面，探讨编译优化如何通过全局视角实现更系统化的优化。
